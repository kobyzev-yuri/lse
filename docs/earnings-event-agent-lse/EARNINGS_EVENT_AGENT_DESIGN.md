# Дизайн: earnings / event agent для LSE

**Версия:** 1.3  
**Дата:** 2026-05-11  
**Контекст:** проект LSE (PostgreSQL + pgvector, `knowledge_base`, `quotes`, `premarket_daily_features`, `trade_history`, GAME_5M, портфельная игра, `AnalystAgent`, LLM). Документ задаёт терминологию для event-контура **над** интрадей-играми; нижележащие документы (GAME_5M, портфель) используют те же английские идентификаторы в коде/JSON и при необходимости поясняются по этому словарю.

## 1. Проблема

Текущая ML-ветка, заточенная под **автоторговлю по одному тикеру**, плохо покрывает задачи инвестора:

- не видит **полный контекст события** (earnings, guidance, commentary, утечки, кросс-влияние);
- слабо формализует **сценарии реакции** (pullback vs переоценка, follow-through vs fade);
- не использует системно **режим рынка** и **синхронность групп**;
- цель «успеть прочитать отчёт и открыть позицию в моменте» для LSE **не первична** — нужен **прогноз типа реакции и зон**, а не только направление.

Нужен отдельный контур **event intelligence**, который может кормить бота/веб аналитикой и опционально — флагами для GAME_5M, но **не заменяется** расширением одной торговой модели.

## 2. Цели

1. **Классификация сценария** после earnings (и аналогичных событий): pullback, смена коридора, follow-through, fade, кросс-заражение peers.
2. **Кросс-отчёты:** влияние отчёта/новости тикера A на группу B (сектор, supply chain, тема).
3. **Циклы и фазы:** вертикальный рост, провал и выкуп, групповой цикл (например AI-chips).
4. **Синхронность / ротация:** кто лидер, кто laggard, куда уходит риск при продаже группы.
5. **Режим индекса:** NDX / SPY / Dow + VIX + breadth — множители и veto для идей по отдельным именам.

Все оценки доходности и обучение — на **log-returns**; симуляции входа/выхода — с **transaction costs** (правила проекта).

| Компонент LSE | Роль для event agent |
|---------------|----------------------|
| `knowledge_base` (`event_type`, `outcome_json`, embedding) | Хранение событий, итогов реакции, RAG похожих кейсов |
| [DATABASE_SCHEMA.md](../DATABASE_SCHEMA.md) | `quotes`, `premarket_daily_features`, `trade_history` |
| [NEWS_SIGNAL_ARCHITECTURE.md](../NEWS_SIGNAL_ARCHITECTURE.md) | Этап A (интерпретация) / B (политика): event agent ближе к **расширению этапа A** под структурированные события earnings + отдельный **event_fusion** перед GAME_5M |

## 4. Данные и схема (предлагаемая)

### 4.1.1 DDL в репозитории

Таблицы **`earnings_event_detail`**, **`peer_graph_edge`**, **`market_regime_daily`**, **`event_reaction_dataset`**: SQL `scripts/sql/ml_event_analytics_schema.sql`, применение `python scripts/migrate_ml_event_analytics.py`. Skeleton-строки датасета из KB: `scripts/build_event_reaction_dataset.py --from-kb-earnings`.

### 4.1 Расширение событий в KB или отдельная таблица

Минимум полей для записи типа `EARNINGS` (имена — для обсуждения):

- `symbol`, `event_time_et`, `fiscal_period`
- `eps_actual`, `eps_estimate`, `revenue_actual`, `revenue_estimate` (или ссылка на внешний факт)
- `guidance_summary` (структурированный JSON)
- `affected_tickers[]` с весами (peers, supply chain)
- `outcome_json`: реакция 1d/2d/5d/20d (log-returns), max drawdown, rebound, volume z-score, `final_label` (сценарий)

Дедуп и источники — в духе существующих миграций `db/knowledge_pg/`.

### 4.2 Датасет `event_reaction_dataset` (логическое имя)

Физическая таблица **`event_reaction_dataset`** (колонки `features_before`, `outcomes_after`, `final_label`, …) создаётся миграцией §4.1.1. Строка = одно событие + окно признаков **до** события + исходы **после**:

- дневные/внутридневные агрегаты из `quotes`, `premarket_daily_features`
- корреляции / beta к NDX, к группе
- признаки «фазы цикла» (дистанция до MA, drawdown from high, vol regime)
- peer confirmation (синхронное движение группы)

### 4.2.1 Feature builder: смысл, содержимое JSONB, хранение в PostgreSQL

**Политика LSE:** обучающий и аналитический контур event-ML держит **источник правды в БД**. Таблица **`event_reaction_dataset`** — основная материализация строки «событие + X до + y после»; выгрузки в CSV допустимы как **экспорт для ноутбуков/внешних тулов**, но не как единственное хранилище признаков.

**Feature builder** — это не отдельная таблица, а **логический слой / код** (крон или скрипт), который для якоря `(symbol, event_time_et)`:

1. **Читает из PostgreSQL** нормализованные сущности: `quotes` (дневные ряды), `premarket_daily_features` (при необходимости срез на дату/лейбл окна события), строки `market_regime_daily` на торговую дату сессии, рёбра `peer_graph_edge` и цены peer’ов для **peer confirmation**, при необходимости — факты из `earnings_event_detail` / текстовый контекст по `knowledge_base_id` (для RAG и LLM отдельно, не обязательно дублировать весь текст внутри `features_before`).
2. **Вычисляет** числовой и категориальный вектор «как рынок и тикер выглядели **до** момента события»: кумулятивные log-returns, волатильность, дистанции до MA, drawdown от хая, заготовки корреляций/beta к бенчмарку (NDX и т.д.), агрегаты по группе аналогов, снимок `ticker_price_regime` (§4.5) на последнюю доступную дату **не позже** якоря события.
3. **Сериализует** результат в **один JSON-объект** и записывает в колонку **`features_before`**. Рекомендуется внутри JSON иметь служебный блок версии, например `feature_builder_version` (строка) и `as_of_trade_date` / `built_at_utc`, чтобы отличать поколения признаков без немедленной миграции столбцов.
4. **Отдельным проходом** (после наступления горизонта наблюдения) заполняется **`outcomes_after`**: log-returns на 1d/2d/5d/20d, max drawdown, rebound, volume z-score и т.п.; человеческая или rule-based метка сценария → **`final_label`** (§6).

**Как используется:**

| Потребитель | Действие |
|-------------|----------|
| **ML head (CatBoost и др.)** | Читает строки из `event_reaction_dataset` (фильтр по `dataset_version`, наличию непустых `features_before` / `outcomes_after`); при необходимости джойнит только `knowledge_base` для доп. полей. Повторный JOIN «сырых» таблиц на каждый эксперимент не обязателен, если builder актуален. |
| **Сервис инференса / бот** | Берёт свежесобранный `features_before` + вероятности сценариев из модели; отдаёт в UI/Telegram вместе с `event_fusion` и лимитами риска. |
| **Контроль качества** | `run_ml_data_quality_report` и метрики по БД: доля строк с непустым `features_before`, покрытие `outcomes_after`, разметка `final_label`. |
| **RAG / LLM** | Embedding и текст остаются в **`knowledge_base`**; `features_before` — структурированный числовой контекст для ML и для «склейки» с ответом LLM. |

Пайплайн: **Ingest** создаёт/обновляет событие в KB (и при необходимости skeleton в `event_reaction_dataset`) → **Feature builder** обновляет `features_before` → по мере зрелости окна — **outcome writer** обновляет `outcomes_after` и `final_label`.

### 4.3 `peer_graph` / тематические кластеры

Граф или таблица рёбер: `(source, target, relation_type, weight)` — усиление существующих `TICKER_GROUPS` / корреляционного контекста GAME_5M на **дневном** горизонте.

### 4.4 `market_regime_daily`

Снимок по дате: режим для SPY/NDX/VIX, breadth (если есть источник), флаги calendar/geo risk (как в [NEWS_SIGNAL_ARCHITECTURE.md](../NEWS_SIGNAL_ARCHITECTURE.md) §6).

### 4.5 `ticker_price_regime` (дневной фон тикера)

Строковая метка **фазы цены** по тикеру (отдельно от сценария одного earnings). Используется вместе с `market_regime_daily` и сценариями §6. Принятые значения (идентификаторы в коде — на английском):

| Идентификатор | Русская формулировка |
|---------------|---------------------|
| `sideways_range` | боковик, диапазонная фаза (типичные качели без устойчивого структурного тренда) |
| `rerate_up` | активный ререйтинг вверх (фаза переоценки, сильный тренд / расширенная волатильность) |
| `derate_down` | обратный ререйтинг вниз (сползание или длинная нога вниз, снижение оценки) |
| `unknown` | режим не классифицирован (мало данных или конфликт признаков) |

#### Классификатор тенденций (черновик порогов, v0)

Идентификаторы в коде/JSON — **английские** (см. таблицу выше). Ниже — **текущие критерии v0**: их можно и нужно **подкрутить** по бэктесту, разметке окон (как у шефа) и стабильности по тикерам. Все доходности — **дневные log-returns** по ряду цен закрытия (adjusted при наличии), согласованно с остальным LSE.

**Входные ряды:** последняя доступная дата `as_of`, дневные закрытия `quotes` (или агрегат из того же источника, что портфель/GAME_5M).

**Признаки (на `as_of`):**

| Обозначение | Описание |
|-------------|----------|
| `r_20` | сумма дневных log-returns за **20** торговых сессий включая `as_of` |
| `r_60` | сумма за **60** сессий |
| `rv_20` | оценка волатильности: `stdev(daily log-ret за 20 дней) × √252` (годовая) |
| `rv_20_pctile_252` | перцентиль текущего `rv_20` относительно **скользящих** окон 20d по истории тикера за последние **252** сессии (если истории мало — см. `unknown`) |
| `dd_60` | **log**-просадка от максимума `log(close)` за 60 сессий до `as_of`: `log(C / max_{60}(C))` ≤ 0 (чем ниже, тем глубже падение от локального хая) |
| `ma20_gt_ma60` | `MA20(close) > MA60(close)` на `as_of` (простые MA по закрытиям) |

**Минимум данных:** для применения правил v0 нужно ≥ **60** валидных дневных закрытий до `as_of` и достаточно истории для `rv_20_pctile_252` (практически ≥ **252** сессий; иначе класс **`unknown`** или упрощённая ветка без перцентиля — на усмотрение реализации, но в отчёте помечать `classifier_note`).

**Порядок присвоения класса (каскад, первое сработавшее правило):**

1. **`unknown`** — если меньше **60** закрытий; или при политике **строгой** к перцентилю не удалось посчитать `rv_20_pctile_252` (`classifier_note: no_vol_percentile`); или мало истории для перцентиля (`insufficient_bars`). *Зарезервировано:* при будущих версиях (score-based, дополнительные оси) — неоднозначность медведь/бык → `classifier_note: conflict_bear_bull` (в **v0** при каскаде п.2→п.3 такой конфликт **не возникает**: медвежьи правила имеют приоритет).
2. **`derate_down`** — обратный ререйтинг / медвежья структура, если выполняется **хотя бы одно**:
   - `dd_60 ≤ log(0.88) ≈ −0.128` (просадка от 60d high не слабее ~**12%** в простых процентах);
   - или `r_20 ≤ −0.04` (≈ **−4%** суммарно за 20 дней в log-шкале);
   - или (`r_60 ≤ −0.08` **и** не `ma20_gt_ma60`) — затяжное снижение и цена ниже «короткой» средней относительно «длинной».
3. **`rerate_up`** — активный ререйтинг вверх, если **ни одно** условие п.2 не выполнено и выполняется **хотя бы одно**:
   - `r_20 ≥ 0.05` (≈ **+5%** за 20 дней);
   - или (`r_60 ≥ 0.10` **и** `r_20 > 0`);
   - или (`r_20 ≥ 0.03` **и** `rv_20_pctile_252 ≥ 0.70`) — умеренный тренд вверх при **повышенной** относительной волатильности.
4. **`sideways_range`** — во всех остальных случаях (нет сильного медвежьего/бычьего сигнала v0): типичный боковик/диапазон относительно выбранных порогов.

**Выход JSON (рекомендуемые поля для отладки и калибровки):** `ticker_price_regime`, `classifier_version: "ticker_price_regime_v0"`, снимок `{ r_20, r_60, rv_20, rv_20_pctile_252, dd_60, ma20_gt_ma60 }`, при `unknown` — `classifier_note` (причина: `insufficient_bars` | `no_vol_percentile` | `conflict_bear_bull`).

**Калибровка:** подбор порогов (−0.04 / +0.05 / 12% / перцентиль 70 и т.д.) — по размеченным интервалам и по распределению ложных срабатываний на портфеле и на `TICKERS_FAST`; изменения порогов оформлять сменой `classifier_version` и строкой в changelog документа.

## 5. Слои системы

1. **Ingest:** earnings facts + KB новости + (опц.) **transcript snippets** (фрагменты транскрипта звонка с инвесторами) → нормализованное событие (в т.ч. строка/skeleton в `event_reaction_dataset`, см. §4.2.1).
2. **Feature builder:** по данным из PostgreSQL формирует и **записывает** JSON **`features_before`** (и опционально денормализованные поля `ticker_price_regime`, `market_regime_date`) для обучения и инференса; см. §4.2.1.
3. **ML head:** вероятности сценариев и/или квантили log-return (калибровка обязательна); обучение от строк `event_reaction_dataset` в БД.
4. **LLM (ограниченно):** извлечение фактов из текста, summary guidance, список конфликтующих фактов; **не** сырой торговый приказ без числового слоя.
5. **RAG:** поиск top-k похожих событий по embedding + фильтр по сектору/режиму.
6. **Вывод:** фиксированный JSON для UI/Telegram: сценарии, вероятности, уровни, invalidation, горизонт, связанные тикеры.

## 6. Классы сценариев (MVP)

- `beat_selloff_pullback` — хороший отчёт, просадка как откат в тренде
- `beat_revaluation_down` — смена мультипликатора / доли в портфелях
- `miss_or_guide_breakdown` — реальное ухудшение тезиса
- `gap_up_follow_through` — продолжение после позитива
- `gap_up_fade` — позитив уже в цене, слабая реакция
- `cross_earnings_contagion` — движение группы от события одного эмитента

Метрики качества — **отдельно по классу** (precision / recall / калибровка вероятностей), не только accuracy.

## 7. Интеграция с GAME_5M

- Event agent выдаёт **флаги** (`earnings_risk`, `no_chase`, `peer_stress`) и **режим**, а не заменяет `recommend_5m`.
- При конфликте: приоритет **риск-лимитов** и режима индекса ([RISK_MANAGEMENT.md](../RISK_MANAGEMENT.md)).

## 8. Этапы внедрения

1. Разметка 50–100 исторических событий (в т.ч. примеры из «Задачи для агента»).
2. DDL для `earnings_event` / расширение KB + `outcome_json` пайплайн backfill.
3. Скрипт сборки `event_reaction_dataset` из БД и внешних фактов.
4. Baseline ML + отчёт по калибровке.
5. LLM extractor по схеме JSON + кэш по `batch_hash` (как в новостном сигнале).
6. Команда бота / раздел веб — **аналитический отчёт**, не обязательный автовход.

В коде, JSON и именах полей БД сохраняются **английские** термины; в отчётах для людей допускается русская формулировка из второго столбца.

| English / идентификатор | Русский аналог (смысл) |
|-------------------------|-------------------------|

| **ticker_price_regime** | дневной режим/фаза цены по тикеру (см. §4.5) |
