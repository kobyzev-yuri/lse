# Почему /recommend5m и /prompt_entry game5m дают разные выводы

При почти одновременных запросах (например 20:29 и 20:31) отчёты могут сильно отличаться: разный вердикт по тикеру (например SNDK — BUY в одном и HOLD в другом), разные RSI, разный период данных и число баров.

## Причины расхождений

### 1. Разный период данных (главная причина)

| Команда | Вызов | Период 5m |
|--------|--------|-----------|
| **/recommend5m** | `get_decision_5m(tkr, days=5, ...)` | последние **5** календарных дней (по умолчанию; можно задать аргументом) |
| **/prompt_entry game5m** | `get_decision_5m(tkr, use_llm_news=True)` без `days` | последние **7** дней (`MAX_DAYS_5M` в `get_decision_5m`) |

Разный `days` даёт:

- разное число баров (например 283 при 5 днях vs 438 при 7);
- разный период в отчёте: «12.03 09:30 – 17.03 13:30» vs «10.03 09:30 – 17.03 13:25»;
- другой расчёт RSI и остальных индикаторов (другой хвост цен).

### 2. Разное правило по новостям для SNDK

В `get_decision_5m` при **позитиве в KB** и пограничном RSI (38–52) решение HOLD может быть сменено на BUY:

```python
elif recent_positive and decision == "HOLD" and rsi_5m is not None and 38 <= rsi_5m <= 52 and momentum_2h_pct >= -0.5:
    decision = "BUY"
    reasons.append("позитив в новостях поддерживает вход при нейтральном RSI")
```

- При **7 днях**: больше баров, RSI может попасть в 38–52 (например 44.4) → срабатывает правило → **BUY**, в отчёте «Влияние новостей: позитив (поддержка входа)».
- При **5 днях**: другой хвост → другой RSI (например 63.9), уже вне 38–52 → правило не срабатывает → **HOLD**, «Влияние новостей: нейтрально».

Итог: один и тот же тикер в один вечер — BUY в одном отчёте и HOLD в другом из‑за разного окна данных и разного RSI.

### 3. Разное время среза (20:29 vs 20:31)

`fetch_5m_ohlc` использует `datetime.utcnow()`, поэтому два запроса с разницей в 2 минуты могут получить разный последний бар от Yahoo (например 13:25 vs 13:30 ET). Это усиливает расхождения в цене, RSI и вердикте.

### 4. Неопределённость LLM

Блок «С учётом корреляции (LLM)» генерируется с temperature > 0; при разном контексте (разные цифры по тикерам) формулировки и акценты закономерно различаются, даже если итоговая рекомендация (HOLD/осторожность) похожа.

## Что сделано для согласованности

- В **prompt_entry game5m** при формировании отчёта по кластеру 5m теперь в `get_decision_5m` передаётся тот же **days=5**, что и по умолчанию в **/recommend5m** (если пользователь не задал другое значение в recommend5m). Так оба отчёта строятся по одному и тому же окну данных (5 дней), и расхождения уменьшаются.

## Единый источник правды и выводы

- **Источник решения 5m:** `services/recommend_5m.get_decision_5m()` — единственное место правил (RSI, импульс 2ч, KB-новости, сессия). Его используют:
  - **Крон** `scripts/send_sndk_signal_cron.py` — вход/выход в игру и рассылка в Telegram.
  - **Бот:** `/recommend5m` (компактная таблица), `/signal5m` (только технический срез), `/prompt_entry game5m` (полный отчёт), `/signal <ticker>` для тикеров из игры 5m.
- **Технический срез** для signal5m и отображения «только техника» — подмножество полей из `get_decision_5m` (константа `TECHNICAL_SIGNAL_KEYS`, функция `get_5m_technical_signal`). Крон по-прежнему вызывает только `get_decision_5m`; расхождений с ботом по логике нет.

## Рекомендации

- Для максимально близких выводов: вызывать одну команду или обе с одинаковым интервалом (например всегда 5 дней) и по возможности в одно время.
- Для сравнения «как было»: в **/recommend5m** можно явно указать период, например `/recommend5m 7` — тогда окно будет 7 дней, как раньше у prompt_entry по умолчанию.
