# Карточки 5m для просмотра в Telegram

Веб-страница с карточками по каждому тикеру игры 5m, удобная для открытия в Telegram (in-app browser): компактный мониторинг, технические параметры, опционально — вывод LLM по кнопке и мини-графики.

---

## Цели

- **Один экран на тикер:** карточка с ключевыми параметрами для принятия решения.
- **Upside / Downside / «Когда дроп»:** на основе существующих полей 5m (тейк, стоп, откат от хая, сессия).
- **Понимание «на чём рост/падение»:** краткое техническое обоснование (reasoning) и, по желанию, развёрнутый вывод LLM (как prompt_entry) — **по кнопке**, чтобы не дергать LLM при каждом открытии.
- **Компактность:** карточки скроллятся сверху вниз; при необходимости — встроенный график (премаркет или 5m по сессии).

---

## Единый источник параметров

Все параметры 5m (upside, downside, prob_up/prob_down, RSI, импульс, стоп/тейк и т.д.) считаются **только в одном месте** — в `get_decision_5m()` (`services/recommend_5m.py`). Список полей для отображения задаётся константой `TECHNICAL_SIGNAL_KEYS`; общий payload для веба и Telegram собирается функцией `get_5m_card_payload(d5, ticker)`.

- **Веб (карточки):** `/api/game5m/cards` вызывает `get_decision_5m()` и формирует карточки через `get_5m_card_payload()`.
- **Telegram (recommend5m, signal5m):** те же данные: `get_decision_5m()` → `get_5m_card_payload()` для таблицы; для одного тикера — `get_5m_technical_signal()` (подмножество тех же ключей).
- **Cron (send_sndk_signal_cron):** использует `get_decision_5m()` → `get_5m_card_payload()`, `build_5m_close_context()`; текст алерта — `build_5m_entry_signal_text(d5, ticker, mentions)` из `services/signal_message_5m.py`; контекст входа — `build_full_entry_context(d5)` из `deal_params_5m.py`.
- **Тексты сообщений 5m:** единый модуль `services/signal_message_5m.py`: полный алерт «ВХОД 5m» — `build_5m_entry_signal_text()`; короткая карточка для чата — `build_5m_technical_short_text()`. Их используют cron и бот (Telegram), при необходимости — веб и другие приложения.

Таким образом, данные и форматирование сообщений 5m идут из одних и тех же модулей во всех приложениях.

---

## Данные в карточке (без LLM)

Источник: `get_decision_5m(ticker, days=5)`; для отображения — `get_5m_card_payload(d5, ticker)` (поля из `TECHNICAL_SIGNAL_KEYS`). Подробный контекст по премаркету и окну импульса см. в документе `docs/GAME_5M_PREMARKET_AND_IMPULSE.md`.

**Апсайд и даунсайд** считаются у нас в `get_decision_5m()` и **всегда выводятся** в карточке: на них опираются риск/ревард, мат. ожидание и чек-лист решения (см. раздел «Чек-лист для принятия решения»).

| Блок | Поля | Пояснение |
|------|------|-----------|
| **Решение** | decision, entry_advice, entry_advice_reason | BUY/HOLD/SELL, ALLOW/CAUTION/AVOID |
| **Upside** | estimated_upside_pct_day, suggested_take_profit_price | Наш расчёт: апсайд на день (тейк), цель по цене |
| **Downside** | estimated_downside_pct_day, prob_up, prob_down | Наш расчёт: риск просадки за день %; P(up)/P(down). Стоп (правило) — stop_loss_pct, только если GAME_5M_STOP_LOSS_ENABLED |
| **Прогноз цены** | price_forecast_5m, price_forecast_5m_summary | Квантили p10/p50/p90 на 30/60/120 мин и P(цена>spot); см. `docs/GAME_5M_PRICE_FORECAST.md` |
| **«Когда дроп» / откат** | pullback_from_high_pct, session_high, recent_bars_low_min | Откат от хая сессии %, уровень хая; при приближении к стопу — «риск дропа» |
| **Техника** | price, rsi_5m, momentum_2h_pct, period_str | Цена, RSI, импульс 2ч, период данных |
| **Контекст** | kb_news_impact, market_session.session_phase | Влияние новостей, премаркет/регулярная сессия |
| **График** | опционально | Ссылка на `/api/chart5m/{ticker}` как img или мини-sparkline |

LLM не вызывается при загрузке страницы. По кнопке «Вывод LLM» — запрос к API, который один раз вызывает анализ с корреляциями (аналог prompt_entry по тикеру) и раскрывает блок под карточкой.

## Как именно считаются Upside / Downside / P(up/down)

Все формулы находятся в `services/recommend_5m.py` в функции `get_decision_5m()` (и вспомогательных функциях `compute_5m_features`, `compute_premarket_features`). Ниже — сжатое изложение.

Обозначения (без LaTeX, чтобы документ читался на GitHub):

- `p` — текущая цена (в регулярной сессии — последняя 5m свеча; в премаркете — `premarket_last`).
- `RSI_5m` — RSI по 5m закрытиям.
- `sigma_5m` — волатильность в %: стандартное отклонение лог-доходностей 5m * 100 (поле `volatility_5m_pct`).
- `mom_2h` — импульс в % (поле `momentum_2h_pct`).
- `low_recent` — минимум последних N баров (поле `recent_bars_low_min`).
- `phase` — фаза сессии (`session_phase`: REGULAR, PRE_MARKET и т.п.).

### 1. Upside (апсайд на день) и цель

1. Сначала считается **эффективный тейк** \(\text{take\_pct}\) (в % от цены) в `services/game_5m._effective_take_profit_pct`:
   - базируется на импульсе \(\text{mom}_{2h}\) и конфигах:
     - `GAME_5M_TAKE_PROFIT_MIN_PCT` — минимальный тейк, при слабом импульсе;
     - `GAME_5M_TAKE_MOMENTUM_FACTOR` и потолок `GAME_5M_TAKE_PROFIT_PCT[_<TICKER>]` — при сильном импульсе.
   - по сути:  
     - если \(\text{mom}_{2h}\) маленький → тейк = потолок;  
     - если \(\text{mom}_{2h}\) большой → тейк ≈ \(\text{mom}_{2h} \times \text{factor}\), но не выше потолка.

2. В `get_decision_5m()`:

   - `estimated_upside_pct_day = take_pct` — поле **Апсайд (тейк)** в карточке.
   - если `p > 0`, то `suggested_take_price = p * (1 + take_pct / 100)` — поле **Цель** в карточке.

Иными словами: апсайд = эффективный интрадей-тейк в процентах, цель = текущая цена * (1 + тейк).

### 2. Downside (риск просадки, НЕ стоп)

Downside показывает **оценочный размер дневной просадки**, а не уровень стоп-лосса. Стоп (правило) берётся отдельно из конфига (`GAME_5M_STOP_LOSS_ENABLED`, `GAME_5M_STOP_LOSS_PCT`) и показывается отдельной строкой «Стоп (правило)».

В `get_decision_5m()` (блок «Оценка downside (риск просадки), НЕ стоп-лосс»):

1. База по волатильности и локальному минимуму:

   - `p` — текущая цена (`out["price"]`),
   - `sigma_5m` — `volatility_5m_pct`,
   - `low_recent` — `recent_bars_low_min`.

   Если `low_recent < p`, вычисляем:
   - `d_recent_low = (p - low_recent) / p * 100`

   Базовый риск:
   - `base = max(2.5 * sigma_5m, d_recent_low)` (если данные есть; иначе `base` стартует с 0 и дальше может быть скорректирован).

2. Коррекция по импульсу, фазе и RSI:

   - Если `mom_2h < -1%`:  
     `base *= 1.2` (отрицательный импульс усиливает риск).
   - Если `phase` в {`PRE_MARKET`, `AFTER_HOURS`}:  
     `base *= 1.1` (вне основной сессии ликвидность ниже).
   - Если `RSI_5m <= 25` (сильная перепроданность):  
     `base *= 0.85` (downside обычно меньше).
   - Если `RSI_5m >= 75` (перекупленность):  
     `base *= 1.15`.

3. Ограничения и итог:

   - минимальный риск: **0.2%**, максимальный: **25%**:  
     если `base > 0`, то `estimated_downside_pct_day = round(min(max(base, 0.2), 25.0), 2)`, иначе — `None`.

В карточке это поле отображается как **«Даунсайд (риск)»**; если в конфиге `GAME_5M_STOP_LOSS_ENABLED=true`, дополнительно показывается **«Стоп (правило)»** из `stop_loss_pct`, чтобы было видно различие между **оценкой риска** и **жёстким правилом стопа**.

### 3. P(up/down) — грубая вероятность направления

Это не статистически откалиброванная вероятность, а **эвристический скаляр** для быстрого сравнения тикеров внутри кластера. Конструкция симметричная: \(\text{prob\_up} + \text{prob\_down} = 1\).

1. Начальные веса:
   - `up_score = 1.0`, `down_score = 1.0`.

2. Коррекция по RSI:

   - Если `RSI_5m <= 30`:  
     `up_score += 0.6`
   - Если `RSI_5m >= 70`:  
     `down_score += 0.6`

3. Коррекция по импульсу:

   - Если `mom_2h >= 1.0%`:  
     `up_score += 0.4`
   - Если `mom_2h <= -1.0%`:  
     `down_score += 0.4`

4. Коррекция по высокой волатильности:

   - Если `sigma_5m >= 0.8%`, оба отклонения от 1.0 «подтягиваются к 1» (делаются менее резкими):  
     `up_score = 1.0 + (up_score - 1.0) * 0.7`  
     и аналогично для `down_score`.

5. Нормировка до вероятностей:

   - `s = up_score + down_score`  
   - `prob_up = up_score / s`  
   - `prob_down = down_score / s`  
   В карточке значения округляются до двух знаков.

Практический смысл: при грубом скане кластера видеть, где P(up) явно перевешивает (низкий RSI + положительный импульс) и где риск больше в сторону падения (высокий RSI + отрицательный импульс).

## Расчёт параметров и их взаимосвязь (пример для карточки)

На примере SNDK (цена 705.41) — как связаны поля карточки и что из них выводить для принятия решения.

| Параметр | Формула / логика | Пример (705.41) |
|----------|-------------------|-----------------|
| **Цель (апсайд)** | Цена × (1 + Апсайд%) = suggested_take_profit_price | 705.41 × 1.06 = 747.73 |
| **Стоп (правило)** | Цена × (1 − stop_loss_pct%) — только если стоп включён | 705.41 × (1 − 2.5/100) ≈ 687.77 |
| **Откат от хая** | (session_high − price) / session_high | При хае 743.93 → (743.93−705.41)/743.93 ≈ 5.17% |
| **Импульс 2ч** | (Цена_сейчас − Цена_2ч_назад) / Цена_2ч_назад = momentum_2h_pct | Рост 9.02% за 2 часа |
| **P(up/down)** | Эвристика по RSI, импульсу, волатильности (см. выше) | 0.56 / 0.44 — грубая оценка направления |
| **RSI(5m)** | Стандартный RSI по 5m закрытиям | 60.6 — нейтрально, не перекупленность |

**Ключевые связи:**

- **Риск/Ревард** = Апсайд% / Даунсайд% = 6.0 / 2.5 = 2.4 → прибыль в 2.4× больше риска.
- **Мат. ожидание** = P(up)×Апсайд% − P(down)×Даунсайд% = 0.56×6.0 − 0.44×2.5 = +2.26% → позитивное ожидание.
- **Импульс + откат** — рост 9.02% за 2ч и откат 5.17% от хая дают контекст «импульс → коррекция → возможно продолжение».
- **RSI** — 60.6 не в зоне перекупленности (>70), что допускает движение к цели.

Итог: техуровни (цель/стоп) задают риск/ревард; импульс и откат — контекст; RSI фильтрует перекупленность; P(up/down) и мат. ожидание оценивают привлекательность сделки.

## Чек-лист для принятия решения (5 параметров)

Минимальный набор понятных параметров, по которым можно принять решение о входе (в духе обсуждения с Квеном). Все нужные величины уже есть в карточке или легко выводятся из неё.

| № | Параметр | Порог принятия | Откуда в карточке |
|---|----------|----------------|-------------------|
| 1 | **Риск/Ревард** | ≥ 1:2 | Апсайд (тейк) / Даунсайд (риск) — например 6.0% / 2.5% = 2.4 |
| 2 | **Мат. ожидание** | > 0% | P(up)×Апсайд − P(down)×Даунсайд, например 0.56×6 − 0.44×2.5 = +2.26% |
| 3 | **Импульс / откат** | Импульс > +5% за 1–2ч ИЛИ откат 3–7% от хая | Импульс 2ч, Откат от хая |
| 4 | **Стоп по капиталу** | ≤ 1–2% портфеля на сделку | Задаётся размером позиции (вне карточки; напоминание в правилах) |
| 5 | **Подтверждение** | ≥2 из: RSI 40–70, объём +20%, пробой уровня | RSI(5m); объём/пробой — по возможности из контекста или LLM |

**Быстрое правило:**  
Вход допустим, если **(Риск/Ревард ≥ 1:2)** и **(Мат. ожидание > 0)** и **(стоп ≤ 2% портфеля)** и есть **≥2 подтверждений** (например RSI в 40–70 и сильный импульс или откат 3–7%).

P(up/down), точный % от хая и т.д. уточняют оценку, но не заменяют эти пять пунктов.

## Как считается «Вход» (ALLOW / CAUTION / AVOID)

Поле **Вход** в карточке — это **наш отдельный слой**: фильтр «можно ли входить в принципе», с учётом **новостей (KB)** и **волатильности**, а не расчёт по пяти параметрам Квена (риск/ревард, мат. ожидание, импульс, стоп по капиталу, подтверждение). Оба подхода дополняют друг друга.

### Наша логика (код: `services/recommend_5m.py`, блок «Совет по входу»)

Исходно **ALLOW**. Затем по порядку:

| Условие | Результат | Причина (entry_advice_reason) |
|--------|-----------|-------------------------------|
| Есть новости с sentiment &lt; 0.35 (very_negative) | **AVOID** | Сильный негатив в новостях — вход отложен |
| Волатильность 5m &gt; 1.0% | **AVOID** | Высокая волатильность 5m — вход рискован |
| Есть новости с sentiment &lt; 0.4 (recent_negative) **или** волатильность 5m &gt; 0.6% | **CAUTION** | Негативные новости в базе — осторожность **или** Волатильность 5m … — осторожность |
| Премаркет **и** гэп к закрытию &lt; −2% | **CAUTION** | Премаркет: гэп … — лучше войти после открытия или лимитом |
| Ни одно не сработало | **ALLOW** | Нет явных ограничений на вход |

Новости берутся из **knowledge_base** за те же `days`, что и 5m данные; sentiment — поле `sentiment_score` (0–1). Таким образом, **новости напрямую влияют на Вход**: сильный негатив → AVOID, слабый негатив или высокая вола — CAUTION.

### Согласование с чек-листом Квена

- **Наш «Вход»** отвечает на вопрос: «нет ли явных причин не входить» (новости, вола, премаркет-гэп). Это **фильтр рисков**, а не оценка привлекательности по риск/ревард и мат. ожиданию.
- **Чек-лист Квена** отвечает на вопрос: «привлекательна ли сделка по соотношению риск/ревард, мат. ожиданию, импульсу и подтверждениям». Это **критерии качества сделки**.

Рекомендуемый порядок: сначала смотрим **Вход** (ALLOW / CAUTION / AVOID). При **AVOID** — не входим. При **ALLOW** или **CAUTION** — дальше проверяем пять параметров (риск/ревард ≥ 1:2, мат. ожидание &gt; 0, импульс/откат, стоп по капиталу, подтверждения). То есть: наш «Вход» отсекает нежелательные ситуации (в т.ч. по новостям), а решение «входить или нет по качеству сделки» — по чек-листу Квена, используя апсайд, даунсайд, P(up/down), импульс, откат и RSI из той же карточки.

## API

- **GET /api/game5m/cards?days=5** — список карточек по всем тикерам игры 5m (только технические данные, без LLM). Ответ: `{ "tickers": [...], "cards": [ { "ticker", "decision", "price", "rsi_5m", "momentum_2h_pct", "volatility_5m_pct", "stop_loss_pct", "take_profit_pct", "estimated_upside_pct_day", "suggested_take_profit_price", "pullback_from_high_pct", "session_high", "entry_advice", "entry_advice_reason", "reasoning", "period_str", "kb_news_impact", "session_phase", "premarket_gap_pct", ... } ], "updated_at" }`.
- **GET /api/game5m/card/{ticker}/llm** — по запросу (кнопка в карточке): вывод LLM с учётом корреляций (как в prompt_entry game5m). Ответ: `{ "ticker", "llm_reasoning", "llm_key_factors", "technical_signal" }`. В контекст LLM передаётся также **средняя волатильность за 20 дней** (см. ниже).

### Средняя волатильность за 20 дней (avg_volatility_20)

Она нужна LLM для контекста («умеренная волатильность» vs «выше среднего»). Источник — таблица **quotes** (дневные данные):

- В **quotes** по каждому тикеру и дате хранится `volatility_5` — скользящее ср. кв. откл. цены закрытия за 5 дней (считается в `update_prices_cron.py` при загрузке дневных котировок).
- **Средняя за 20 дней** = AVG(volatility_5) по последним 20 записям в quotes для тикера; в промпт LLM передаётся в % от последней цены (для сопоставимости с волатильностью 5m в %).

**Чтобы данные появились:** по тикеру должны быть заполнены дневные котировки в `quotes` (скрипт `update_prices_cron.py` по расписанию). Если по тикеру нет 20 дней в quotes, поле в контексте LLM будет пустым (N/A). Тикеры игры 5m обычно входят в список обновления цен (TICKERS_FAST / GAME_5M_TICKERS).

## Вёрстка для Telegram

- **Viewport:** mobile-first, узкая колонка (до 420px), чтобы в Telegram WebView всё читалось без горизонтального скролла.
- **Карточки:** одна под другой, скролл сверху вниз. В каждой: заголовок (тикер + решение), блоки «Upside / Downside / Откат», краткое обоснование, кнопка «Вывод LLM»; при необходимости — маленький график (премаркет или 5m).
- **Визуал:** цвет решения (BUY — нейтрально-зелёный, HOLD — серый, SELL — красноватый), иконки или короткие подписи для upside/downside/откат. Без тяжёлых библиотек: минимум JS, при необходимости Chart.js уже есть в base.html.
- **График:** по умолчанию можно не показывать; добавить кнопку «График 5m» или встроить img с `src="/api/chart5m/{ticker}?w=400"` если API отдаёт PNG.

## Реализация

- **Шаблон:** `templates/game5m_cards.html` — отдельная страница (без base), mobile-first, тёмная тема, скролл карточек.
- **Содержимое карточки:** тикер, decision (цветовая полоска BUY/HOLD/SELL), цена, RSI(5m), Upside (тейк % и цена), Downside (стоп %), импульс 2ч, откат от хая, совет по входу, краткое reasoning. Кнопка «Вывод LLM» запрашивает `/api/game5m/card/{ticker}/llm` и раскрывает блок под карточкой. Ссылка «График 5m» ведёт на `/visualization?ticker=...`.
- **Навигация:** в `base.html` добавлена ссылка «5m карточки»; на странице карточек — кнопка «Обновить» и ссылка «Мониторинг».

---

## Итог

- Мониторинговая страница: только технические параметры и выводы по правилам 5m.
- LLM — по кнопке, один запрос на тикер при необходимости.
- Карточки компактные, с интуитивными подписями (upside, downside, откат от хая / риск дропа).
- Удобно открывать в Telegram и скроллить сверху вниз.
