# Стратегии, LLM и отчёты: подходы и реализация

Документ фиксирует подходы к стратегиям, использованию LLM при принятии решений о входе и к отчётам в Telegram-боте.

---

## 1. Стратегии в системе

### 1.1 Источники сделок (кто открыл/закрыл)

| Стратегия   | Описание |
|------------|----------|
| **GAME_5M** | Игра 5m: крон по тикерам из `GAME_5M_TICKERS`, интрадей (вход/выход по 5m, тейк/стоп). В `/pending` для тикеров, убранных из списка, показывается «5m вне» — крон по ним не управляет. |
| **Portfolio** | Портфельный цикл (`trading_cycle_cron`, ExecutionAgent). Сделки по сигналу AnalystAgent по списку MEDIUM/LONG. Если StrategyManager не вернул имя стратегии, в БД пишется «Portfolio». Стоп-лосс проверяется при каждом запуске крона. |
| **Manual** | Ручные команды `/buy` и `/sell` в боте. |

Подробнее: команда бота `/strategies`, см. также `docs/STRATEGY_FACTORY.md`.

### 1.2 Стратегии StrategyManager (портфельный цикл)

Выбираются по режиму рынка (волатильность, sentiment, гэп):

- **Momentum** — низкая волатильность + положительный sentiment.
- **Mean Reversion** — высокая волатильность + нейтральный sentiment.
- **Volatile Gap** — очень высокая волатильность + гэп или экстремальный sentiment.
- **Geopolitical Bounce** — резкое падение предыдущей сессии (≥2%), отскок long.
- **Neutral** — fallback, когда ни одна стратегия не подошла; консервативный HOLD.

**Portfolio** vs **Neutral**: Portfolio — метка по умолчанию при пустом имени стратегии; Neutral — явно выбранная стратегия (режим не определён, удержание).

## 2. LLM при принятии решения о входе

### 2.1 Промпт

- Системный промпт задаёт роль (финансовый аналитик) и формат ответа (JSON: decision, confidence, reasoning, risks, key_factors).
- В пользовательское сообщение передаются: тикер, технические данные (цена, SMA, волатильность, RSI, технический сигнал), sentiment, новости (топ-5), при необходимости — контекст премаркета.

Получить актуальный текст промпта: команда бота `/prompt_entry`. Код: `services/llm_service.py` — `get_entry_decision_system_prompt()`, `analyze_trading_situation()`.

### 2.2 Учёт стратегии в промпте

- В промпт передаётся **выбранная стратегия** (StrategyManager) и **её сигнал** (BUY/HOLD/SELL).
- Строка в промпте: «Контекст стратегии: выбранная стратегия — {name}, её сигнал — {signal}. Учти при итоговой рекомендации (можешь согласиться или скорректировать).»

Реализация: `analyst_agent.get_decision_with_llm()` передаёт `strategy_name` и `strategy_signal` в `llm_service.analyze_trading_situation()`.

### 2.3 Учёт исходов по стратегиям (история)

- Собирается **агрегированная статистика** по закрытым сделкам за последние 180 дней: по каждой стратегии (entry_strategy) — число сделок, число в плюс, доля в плюс (%), суммарный PnL ($).
- Эта строка передаётся в LLM как `strategy_outcome_stats`: «История по стратегиям (закрытые сделки): … Учти исходы в похожих ситуациях при рекомендации.»

Реализация: `report_generator.get_strategy_outcome_stats(engine, limit_days=180)`, вызов из `analyst_agent.get_decision_with_llm()`, параметр `analyze_trading_situation(..., strategy_outcome_stats=...)`.

Ограничение: «похожесть» ситуаций пока не учитывается — передаётся общая статистика по всем закрытым сделкам за период. Фильтрация по режиму (волатильность/sentiment на момент входа) может быть добавлена при появлении сохранения режима при открытии сделки.

### 2.4 Финальное решение

- Если LLM вернул BUY или STRONG_BUY — берётся решение LLM.
- Иначе используется **base_decision** от выбранной стратегии (StrategyManager).

### 2.5 Где используется LLM для прогноза, где нет

Чтобы не путать пользователя разными исходами (с LLM и без), в **Telegram-боте** все команды, показывающие прогноз (решение BUY/HOLD/STRONG_BUY), используют один путь — **с LLM**:

| Команда | Использует LLM |
|---------|----------------|
| `/signal <ticker>` | да |
| `/recommend <ticker>` | да |
| `/ask` (вопрос с тикером, анализ) | да |
| `/prompt_entry <ticker>` | да (тот же analyst) |

Бот создаёт `AnalystAgent(use_llm=True)`. Решение пересматривается моделью по полному контексту (новости, риски, геополитика).

**Без LLM** (только стратегия + тех. сигнал + sentiment, без пересмотра моделью):

| Место | Причина |
|-------|---------|
| **Дашборд** (`/dashboard`, `send_dashboard_cron`) | Свой `AnalystAgent(use_llm=False)` и `get_decision(ticker)` — экономия API при частой рассылке по крону. |
| **Backtest** | По умолчанию `use_llm=False` для скорости. |

Разница: с LLM модель может скорректировать решение (например, HOLD при геополитическом риске при открытых позициях); без LLM остаётся только базовое решение стратегии.

**Управление:** параметр **USE_LLM** в `config.env` (или в глобальных параметрах БД `strategy_parameters`, entity=GLOBAL). При `USE_LLM=false` LLM не применяется (бот и trading_cycle используют только стратегию + тех. сигнал + sentiment). Иначе — применяется. Функция `config_loader.get_use_llm_for_analyst(engine=None)` — единая точка чтения (если передан engine, значение может браться из БД).

### 2.6 Корреляция с бенчмарком и контекст по пулу

**Сейчас (для одной бумаги):** в промпт по тикеру (кроме MU) передаётся скользящая корреляция лог-доходностей с бенчмарком **MU** (14 дн.) и **текущий сигнал по MU** (без LLM). LLM видит, например: «Корреляция с бенчмарком (MU, 14 дн.): 0.72 (In-Sync). Текущий сигнал по MU: BUY. При высокой корреляции движение бенчмарка может поддерживать или давить на тикер.» Реализация: `report_generator.get_rolling_corr_with_benchmark()`, `analyst_agent._get_benchmark_signal("MU")`, блок в `llm_service.build_entry_prompt` / `analyze_trading_situation`.

**Общий случай (идея на будущее):** для игры с несколькими бумагами (например все тикеры из `get_tickers_game_5m()`) имело бы смысл:
- считать **корреляции по всем парам** (или каждой бумаги с «центроидом» пула) и передавать в промпт **матрицу/сводку по пулу**;
- передавать **сигналы по всем тикерам пула** (базовые, без LLM);
- давать LLM **общий контекст по пулу** и либо решение по каждому тикеру с учётом пула, либо отдельный режим «общее решение по пулу» (приоритеты, риски, не открывать всё сразу при высокой корреляции и т.п.).

Пока в игре 5m две бумаги (SNDK, MU), достаточно текущей схемы «тикер + MU». При расширении пула — перейти к контексту по всему пулу.

## 3. Отчёты в Telegram-боте

### 3.1 История сделок (`/history`)

- Маркеры по **фактическому PnL**, а не по signal_type: выход в плюс (цена выхода ≥ цены входа) → синий кружок (тейк), в минус → красный (стоп).
- Легенда: 🟢 Вход · 🔵 Выход в плюс · 🔴 Выход в минус.

### 3.2 Закрытые позиции (`/closed`)

- Колонки **Entry** и **Exit** — стратегия открытия (первый BUY) и стратегия закрытия (SELL), чтобы различать случаи «открыли в GAME_5M, закрыли в Manual».
- Реализация: `report_generator.TradePnL` — поля `entry_strategy`, `exit_strategy`; в `compute_closed_trade_pnls` ведётся `position_open_strategy` при первом BUY по позиции.

### 3.3 Открытые позиции (`/pending`)

- Колонка **Strategy** — стратегия последнего BUY по позиции; для тикеров, убранных из игры 5m, показывается «5m вне» (крон не управляет).
- Колонки **Now** и **P/L** — текущая цена (последняя close из `quotes`) и нереализованный P/L (% и $).
- Цена и P/L считаются по последней доступной дневной close в БД.

### 3.4 Переназначение стратегии

- Команда `/set_strategy <ticker> <стратегия>` и скрипт `scripts/set_position_strategy.py`: обновляют `strategy_name` у **последнего BUY** по тикеру в `trade_history`.
- Используется для позиций «5m вне» (например GC=F после вывода из GAME_5M): переназначение на Manual или Portfolio.

## 4. Параметры конфигурации и возможность менять их без перезапуска

### 4.1 Как сейчас читаются параметры

| Где | Параметры | Когда читаются |
|-----|-----------|----------------|
| **config.env** | Все переменные | При каждом вызове `get_config_value()` заново читается файл (кэша нет). |
| **game_5m** | GAME_5M_STOP_LOSS_PCT, GAME_5M_TAKE_PROFIT_PCT, GAME_5M_MAX_POSITION_DAYS, GAME_5M_TAKE_PROFIT_MIN_PCT, GAME_5M_STOP_TO_TAKE_RATIO, GAME_5M_STOP_LOSS_MIN_PCT | При каждом вызове `get_strategy_params()`, `_effective_take_profit_pct()`, `should_close_position()` — т.е. при каждом запуске крона. Изменение config.env на диске подхватится следующим запуском крона (но не бота, если крон — отдельный процесс). |
| **execution_agent** | STOP_LOSS_LEVEL (5%) | Константа в коде (0.95), не из config. Перезапуск не поможет — только правка кода или вынос в config. SANDBOX_SLIPPAGE_SELL_PCT читается при вызове `_get_slippage_sell_pct()` каждый раз. |
| **StrategyManager** | Пороги выбора стратегии (волатильность 1.5x, sentiment ±0.6, гэп 3%) | Зашиты в `__init__` менеджера, не из config. Создаётся один раз при старте AnalystAgent. |
| **ticker_groups** | GAME_5M_TICKERS, TICKERS_FAST, TICKERS_MEDIUM, TICKERS_LONG | При каждом вызове `get_tickers_game_5m()` и др. — снова читается config. |

Итог: часть параметров (GAME_5M_*, тикеры) по факту читается при каждом обращении к ним, но источник один — config.env на диске. Долгоживущий процесс (бот, воркер) не перечитывает файл, если его править снаружи, пока не вызван код, который снова вызывает `get_config_value`. Пороги StrategyManager и стоп портфеля (STOP_LOSS_LEVEL) в коде — без перезапуска не меняются.

### 4.2 Имеет ли смысл вариации параметров

Да, имеет смысл:

- **Быстрая подстройка** — ужесточить стоп или тейк на период волатильности без деплоя.
- **Эксперименты и A/B** — разные наборы параметров для части тикеров или времени.
- **Агент/пользователь** — LLM или пользователь предлагает «поднять тейк до 6%»; применение без перезапуска и без правки файла на сервере.

### 4.3 Возможный подход: runtime-переопределения

- **Хранилище переопределений** — таблица в БД (например `config_overrides`: key, value, updated_at). При чтении параметра: сначала проверять переопределение, при отсутствии — `get_config_value()` (config.env).
- **Команда бота** — например `/config get [KEY]` и `/config set KEY VALUE` (для доверенных пользователей), чтобы просматривать и менять переопределения. Список ключей можно ограничить (только GAME_5M_*, STOP_LOSS_LEVEL после выноса в config и т.д.).
- **Агент** — может получать текущие значения (из конфига + overrides) и предлагать изменения в ответе; фактическое применение — через `/config set` или отдельный подтверждаемый шаг.

Тогда параметры, которые уже читаются в рантайме (GAME_5M_*, тикеры), начнут учитывать overrides без перезапуска; константы в коде (STOP_LOSS_LEVEL, пороги StrategyManager) для такой схемы нужно вынести в чтение из config/overrides при каждом использовании.

### 4.4 Предпочтительная модель: параметры и история в БД (обсуждение)

Целевая модель: **в базе данных хранятся параметры стратегий и история их изменений**, чтобы потом учитывать эффективность в зависимости от того, какие значения стояли в тот или иной период.

- **Параметры** — текущие значения (тейк/стоп, пороги выбора стратегии, макс. срок позиции и т.д.) хранятся в БД. При принятии решений и при работе крона читается актуальный набор из БД (с fallback на config.env при отсутствии записи).
- **История изменений** — каждое изменение параметра фиксируется: кто изменил (оператор / агент), когда, какое значение было и стало, при необходимости — причина или контекст (например, «повышенная волатильность»). Это даёт временную шкалу: «с такой-то даты по такую-то действовали такие параметры».
- **Учёт эффективности** — по закрытым сделкам и по периодам можно сопоставлять: при каких параметрах какая была доходность, просадка, винрейт. Оператор или агент меняет параметры в зависимости от трендов и режимов рынка; история в БД позволяет оценить, насколько удачными были эти изменения.

Таким образом, БД выступает единым источником правды и по текущим настройкам, и по их эволюции, что нужно для анализа эффективности стратегий при изменениях, вносимых оператором или агентом.
