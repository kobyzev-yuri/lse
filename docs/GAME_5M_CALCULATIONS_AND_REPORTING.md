# Игра 5m (GAME_5M): расчёты тейка/стопа, отчёты и графики

Краткий документ для руководства: как устроена **одна** активная стратегия песочницы **GAME_5M** (только **лонг**), откуда берутся числа в отчётах и на графиках, и как это стыкуется с параметрами из рабочего `config.env` (ниже — значения **в игровом блоке**, как в `config.env.lse` на момент подготовки документа; **секреты и ключи API в документ не включались**).

---

## 1. Что это за «игра»

- Запись сделок в PostgreSQL, таблица `trade_history`, стратегия **`strategy_name = GAME_5M`**.
- Крон: `scripts/send_sndk_signal_cron.py` — по тикерам из **`TICKERS_FAST`** (или узкий список `GAME_5M_TICKERS`, если задан): проверка открытой позиции, закрытие по правилам, при сигнале BUY — запись входа и рассылка (если включена).
- Сейчас **нет отдельной стратегии шорта** в этой игре: решения и позиции трактуются как **лонг**.

---

## 2. Актуальные параметры из `config.env.lse` (игровой блок)

| Параметр | Значение (как в lse) | Смысл |
|----------|----------------------|--------|
| `TICKERS_FAST` | SNDK, NBIS, ASML, MU, LITE, CIEN | Быстрые тикеры; по ним в т.ч. крутится игра 5m |
| `GAME_5M_COOLDOWN_MINUTES` | 120 | Пауза между **рассылками** сигнала на вход по тикеру |
| `GAME_5M_STOP_LOSS_ENABLED` | false | Стоп по убытку **выключен** — автозакрытие по цене в основном тейк / TIME_EXIT / SELL и т.д. |
| `GAME_5M_TAKE_PROFIT_PCT` | 7 | Общий **потолок** тейка (%) |
| `GAME_5M_TAKE_PROFIT_PCT_*` | SNDK 6; NBIS 7.5; ASML 4; MU 4; LITE 6; CIEN 7.5 | Потолок тейка по тикеру (перекрывает общий) |
| `GAME_5M_TAKE_PROFIT_MIN_PCT` | 4.0 | Порог импульса 2ч: **ниже** — тейк = потолок из конфига; **от 4%** — тейк от импульса (см. §3) |
| `GAME_5M_TAKE_MOMENTUM_FACTOR` | *(не задан → в коде 1.0)* | Множитель к импульсу при ветке «тейк от импульса» |
| `GAME_5M_MAX_ATR_5M_PCT` | 1.13 | Порог качества (если используется в цепочке решений) |
| `GAME_5M_MAX_POSITION_DAYS` / `*_SNDK` … | 1; SNDK 3; NBIS 5; … | Максимальное удержание по календарю (и переопределения по тикеру) |
| `GAME_5M_SESSION_END_*` | 30 мин, 0.3% | TIME_EXIT в хвосте сессии при минимальном плюсе |
| `COMMISSION_RATE` | 0 | В этом конфиге комиссия в расчётах **нулевая** |

Источник логики в коде: `services/game_5m.py`, крон: `scripts/send_sndk_signal_cron.py`, импульс 2ч: `services/recommend_5m.py`.

## 3. Как считается порог тейка (упрощённо)

1. **Потолок `cap`** по тикеру: `GAME_5M_TAKE_PROFIT_PCT_<TICKER>` или общий `GAME_5M_TAKE_PROFIT_PCT`.
2. **Импульс `momentum_2h_pct`**: приблизительно прирост цены за окно до ~2 часов по 5m-свечам (в премаркете окно другое — см. код).
3. **Эффективный тейк `take_pct`**:
   - если `momentum_2h_pct >= GAME_5M_TAKE_PROFIT_MIN_PCT` (у вас **4%**):  
     `take_pct = min(momentum_2h_pct × factor, cap)` , где `factor` по умолчанию **1.0**;
   - иначе: **`take_pct = cap`**.
4. **Срабатывание**: сравнивается **не импульс сам с собой**, а **ценовая** нереализованная прибыль от входа: берётся `max(текущая_цена, high_бара)` и проверка `≥ take_pct − 0.05` п.п. (малый допуск на расхождение котировок).
5. **Потолок в конфиге — не «обязательный минимум прибыли»**: при сильной ветке от импульса выход может быть **раньше**, чем `cap` (например при `take_pct ≈ 4–5%` при `cap` 6–7%).

## 4. Примеры из практики отчётов (иллюстрация)

### 4.1. LITE и NBIS (разный фактический % при одной причине TAKE_PROFIT)

- У **LITE** потолок в конфиге **6%**, но эффективный `take_pct` мог быть **меньше** (импульсная ветка после порога 4%), плюс в БД фиксируется **Close** бара, а решение могло опереться на **High** — в отчёте виден **меньший** реализованный %.
- У **NBIS** потолок **7.5%**, сработал тейк по ценовому условию, а **фактическая цена выхода** оказалась выше минимума для триггера — в таблице **реализованный %** может быть **выше** номинального «потолка в таблице настроек» для строки.

Это не противоречие: **колонка причины** — правило срабатывания; **колонка %** — факт `(exit − entry) / entry`.

### 4.2. SNDK: два входа в один день, одна строка в «Открытых»

Наблюдение: на графике 5m — **две** отметки покупок (например около **11:30** и **13:30** ET), в `/reports/pending` — **одна** строка с ценой входа **~850.67**.

Объяснение:

- В БД — **два** `BUY` подряд без полного `SELL` между ними.
- Отчёт **агрегирует** позицию на тикер: **средневзвешенная** цена входа и **суммарный** объём. Для двух покупок **844.63** и **856.71** при одинаковом количестве средняя ровно **850.67**.
- Дата «Open» в отчёте — время **первого** `BUY` текущей «ноги», а не последнего.
- После доработки UI: колонка **`BUYs`** = число покупок в текущей открытой позиции (**2** в этом сценарии).

Это **не** модель «два независимых игрока»: это **один** усреднённый счёт на тикер до полного закрытия. Несколько входов одним участником на реальном рынке **разрешены**; в симуляторе выбрана упрощённая агрегация.

---

## 5. FIFO vs средняя (для закрытия)

- При **полном** закрытии одним `SELL` **итоговый** P/L в долларах совпадает у учёта по **средней** и у **FIFO** по лотам — меняется только «раскладка» по частичным выходам, если бы они были.
- Текущая игра чаще закрывает **весь** остаток одной продажей — средняя удобна для одной строки в pending и одного порога тейка.

## 6. Графики и отчёты: что изменилось в интерфейсе

### 6.1. `/reports/pending`

- Пояснение под заголовком: одна строка на тикер = **средневзвешенный** вход; дата — **первый** `BUY`; на графике каждый `BUY` из истории за период может быть отдельной точкой.
- Колонка **`BUYs`**: сколько покупок вошло в текущую открытую позицию (до полного `SELL`).

### 6.2. Визуализация 5m (`templates/visualization.html`)

- Вместо текстовых `*` у сделок — **треугольники**: вход **▲** (зелёный); закрытие в плюс к **средней на момент этого SELL** — **▲** зелёный; в минус — **▼** красный; без базы для расчёта — серый **▲**.
- API `/api/chart5m` отдаёт по сделкам также **`quantity`** и **`id`** для корректного учёта при нескольких лотах на графике.

### 6.3. Telegram `/chart5m` и общий график игры

- Та же логика классификации выходов по знаку P/L к средней на момент продажи (`partition_trades_for_chart_pnl` в `services/game_5m.py`).

---

## 7. Ограничения и направление развития

- Сейчас: **одна** стратегия **GAME_5M**, **лонг**. Другие стратегии или «независимые игроки» возможны через разные `strategy_name`, отдельные списки тикеров и разделение ключа позиции **`(strategy_name, ticker)`** — это отдельный объём разработки.
- Шорт как **вторая независимая игра** возможен только как отдельная политика (например отдельное имя стратегии), а не как автоматическое следствие текущего кода.
