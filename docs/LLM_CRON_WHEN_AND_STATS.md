# Когда включать LLM в cron и какая статистика нужна

Кратко: когда оправдано вызывать LLM по расписанию (а не только по запросу пользователя) и чего не хватает для обоснованного решения.

---

## Текущая схема

| Где | Кто решает | LLM |
|-----|------------|-----|
| **Cron (send_sndk_signal_cron, premarket и т.д.)** | Теханализ: `get_decision_5m(..., use_llm_news=False)` — правила по RSI, импульсу, стоп/тейк, KB. Вход/выход по ним. | Не вызывается (или только опционально «новости» через `use_llm_news`, без полного анализа). |
| **По запросу пользователя** | Те же правила + при необходимости полный анализ LLM. | В Telegram (`/recommend5m`, `/prompt_entry game5m`) и на веб (кнопка «Вывод LLM») вызывается `analyze_trading_situation`: один запрос на тикер/кластер, токены только когда пользователь просит. |

Итого: **решения по входу/выходу в cron принимает теханализ**; LLM даёт пояснения и опциональную корректировку **только по запросу**, без расхода токенов по расписанию.

---

## В каком случае было бы оправдано включать LLM в cron

Два разумных варианта:

### 1. Геополитический риск (удержание / превентивный выход)

- **Идея:** при появлении в новостях (KB) признаков эскалации — LLM в cron мог бы давать рекомендацию «HOLD / рассмотреть закрытие позиций», даже если теханализ даёт BUY.
- **Плюс:** один вызов LLM раз в N часов (например раз в 4–6 ч или только в определённые часы/дни) по кластеру или по «гео-контексту», а не по каждому тикеру каждые 5 минут — расход токенов ограничен.
- **Минус:** без статистики непонятно, как часто LLM действительно предотвращает потери в такие периоды и не даёт ли лишних ложных HOLD.

**Вывод:** включать LLM в cron **только для геополитики** может быть оправдано, если:
- есть явный сценарий (например «пятница + ключевые слова в KB → один вызов LLM по контексту → рекомендация держать/закрыть»);
- и дальше ведётся учёт: в какие дни LLM переключил решение на HOLD/выход и какой был фактический результат (PnL, просадка) по тем позициям.

### 2. Более широкий контекст при неоднозначном теханализе

- **Идея:** когда правила дают пограничный сигнал (например RSI перепродан, но импульс слабый), раз в период вызывать LLM «по кластеру» и использовать его вывод как доп. фильтр или оверрайд.
- **Минус:** многократные вызовы по крону (каждые 5–15 мин по нескольким тикерам) быстро съедают токены; польза без замеров неочевидна.

**Вывод:** для «широкого контекста» в cron включать LLM **не оправдано**, пока нет доказательства, что это улучшает исходы по сравнению с голым теханализом. Лучше оставить LLM по запросу (Telegram / веб).

---

## Чего не хватает: статистика «LLM vs теханализ»

Чтобы обоснованно решить, подключать ли LLM в cron (хотя бы для гео-риска), нужны сравнимые исходы:

1. **A/B по исходам сделок**
   - Для части тикеров/периодов логировать: какое решение было бы по теху, какое дал бы LLM (можно вызывать LLM офлайн по сохранённому контексту).
   - Считать по закрытым сделкам: если бы следовали только теху vs если бы следовали LLM (оверрайд) — PnL, число выходов в минус, просадка.
   - Метрики: win rate, средний PnL на сделку, макс. просадка за период, отдельно по периодам с геополитическими новостями.

2. **Бэктест (офлайн)**
   - Исторические данные: 5m, KB, при желании — снимки новостей за прошлые дни.
   - Два сценария: (A) решение только по правилам 5m; (B) решение по правилам + оверрайд LLM (например «при гео-ключевых словах в KB вызывать LLM раз в день и при HOLD от LLM не открывать / рекомендовать закрыть»).
   - Сравнить кривую капитала, число сделок, просадки в даты с гео-событиями.

3. **Минимальный шаг без полного бэктеста**
   - Логировать в БД или лог: дата/время, тикер, техническое решение, текст из KB (или теги), и при ручном/редком запуске — ответ LLM (BUY/HOLD/SELL + краткий reasoning).
   - Периодически разбирать вручную: в случаях, где LLM бы изменил решение (оверрайд), что произошло бы по цене на следующий день/неделю. Так можно накопить качественную выборку «LLM был бы прав / не прав» перед автоматизацией в cron.

---

## Рекомендация

- **Сейчас:** не включать полный LLM в cron для каждого тикера/каждые 5 минут — токены растут, выгоды без статистики нет.
- **По запросу (Telegram, веб)** — оставить как есть: пользователь сам решает, когда запросить вывод LLM.
- **Геополитика:** если хотите пробовать LLM в cron только для «удержания по гео-риску» — сделать один вызов LLM редко (например раз в день или в пятницу по контексту KB) и писать результат в лог/алерт; параллельно завести учёт «дни с гео-оверрайдом LLM» и последующий PnL/просадку.
- **Статистика:** перед тем как включать LLM в cron «на постоянной основе», добавить логирование технического решения и (где уже вызывается LLM) решения LLM по одному и тому же контексту, затем считать сравнение исходов (хотя бы по закрытым сделкам и по датам с гео-новостями). Имеет смысл опираться на документ [GEOPOLITICAL_RISK_AND_CAUTION.md](GEOPOLITICAL_RISK_AND_CAUTION.md) для уровней риска и формулировок.
