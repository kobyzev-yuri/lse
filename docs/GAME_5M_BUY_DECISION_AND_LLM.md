# Как формируется решение BUY в игре 5m: входные параметры и роль LLM

Детальное описание цепочки от данных до записи входа в `trade_history`: что входит в расчёт, где участвует LLM, что сохраняется в `context_json`.

---

## 0. Две стратегии входа

| Стратегия | Описание | Использование |
|-----------|----------|----------------|
| **Технический вход** | Решение только по правилам 5m + новости KB + фаза сессии. Без корреляций и без LLM. | Текущая по умолчанию. Запись в `trade_history` по `get_decision_5m()`. |
| **LLM-вход** | Те же данные + контекст корреляций и связанных тикеров; решение принимает LLM (BUY/HOLD/SELL). | Перспектива для тестирования. Включение: `GAME_5M_ENTRY_STRATEGY=llm`. |

Переключатель в config: **`GAME_5M_ENTRY_STRATEGY`** = `technical` (по умолчанию) | `llm`. При `llm` крон перед записью входа запрашивает LLM с контекстом корреляций; в БД пишется решение LLM только если оно BUY/STRONG_BUY. Это позволяет сравнивать результаты технического и LLM-входа (например, по /closed и /closed_impulse).

---

## 1. Общая схема

```
5m OHLC (Yahoo) + KB новости + сессия биржи (+ опционально LLM-новости)
        ↓
  get_decision_5m(ticker, use_llm_news=True)
        ↓
  Правила (RSI, импульс, волатильность) → базовый decision
        ↓
  Коррекция по новостям KB (негатив/позитив) → итоговый decision
        ↓
  Сессия (PRE_MARKET/AFTER_HOURS → HOLD)
        ↓
  d5 = { decision, reasoning, price, momentum_2h_pct, ... }
        ↓
  Крон: если decision in (BUY, STRONG_BUY) и нет позиции, сессия REGULAR, cooldown пройден
        ↓
  record_entry(ticker, price, decision, reasoning, entry_context=build_full_entry_context(d5))
```

**Важно:** В кроне (`send_sndk_signal_cron.py`) решение о BUY **не** проходит через Analyst LLM (`get_decision_with_llm`). Итоговый `decision` даёт только `get_decision_5m()`: правила + новости KB + фаза сессии. LLM участвует в двух других местах (см. раздел 5).

---

## 2. Входные данные для решения (get_decision_5m)

### 2.1 5m-свечи (Yahoo Finance)

- **Источник:** `fetch_5m_ohlc(ticker, days=7)` → до 7 дней 5-минутных OHLC.
- **Используются:**
  - `price` — последний Close.
  - `high_5d`, `low_5d` — макс. High и мин. Low за окно.
  - `last_bar_high`, `last_bar_low`, `recent_bars_high_max`, `recent_bars_low_min` — последний бар и экстремумы за последние 6 баров (30 мин) для тейка/стопа.
  - `session_high` — хай текущей торговой сессии (последний день в данных).
- **Производные:**
  - **RSI(5m)** — по 14 свечам (`compute_rsi_5m`).
  - **Волатильность 5m** — стандартное отклонение лог-доходностей по окну, в %.
  - **Импульс 2ч** — `momentum_2h_pct = (price / close_24_bars_ago - 1) * 100` (реализованное изменение за ~2 часа).
  - **Кривизна** — вторая разность последних закрытий (ускорение цены).
  - **Откат от хая** — `pullback_from_high_pct`, возможный отскок до `session_high`.
- **Период:** `period_str` (диапазон дат/времени баров).

### 2.2 Новости из базы знаний (KB)

- **Источник:** `fetch_kb_news_for_period(ticker, days)` — новости по тикеру за тот же период, что и 5m.
- **Использование:** по каждой новости берётся `sentiment_score`; агрегируются флаги:
  - `very_negative` — хотя бы одна новость с sentiment < 0.35;
  - `recent_negative` — sentiment < 0.4;
  - `recent_positive` — sentiment > 0.65.
- В решении используется только эта агрегация; полные тексты в решение не подставляются, но передаются в `kb_news` и `kb_news_impact` для вывода в алерт/отчёт.

### 2.3 Сессия биржи

- **Источник:** `get_market_session_context()` → `session_phase` (REGULAR, PRE_MARKET, NEAR_OPEN, NEAR_CLOSE, AFTER_HOURS, WEEKEND, HOLIDAY), при необходимости `minutes_until_open` / `minutes_until_close`.
- В **премаркете** дополнительно вызывается `get_premarket_context(ticker)`: цена премаркета, гэп к закрытию, минуты до открытия; это попадает в `reasoning` и в рекомендации по лимитам.

### 2.4 Опционально: LLM-новости (не меняют decision)

- **Условие:** `use_llm_news=True` и в config: `GAME_5M_USE_LLM_NEWS=true`, `USE_LLM_NEWS=true`.
- **Действие:** запрос к `llm_service.fetch_news_for_ticker(ticker)` → в `d5` добавляются `llm_news_content`, `llm_sentiment`, `llm_insight` (и при наличии `llm_comparison`).
- **Роль:** только для отображения в алерте и в отчётах; **логика BUY/HOLD/SELL в get_decision_5m от этих полей не зависит**.

## 3. Правила решения (get_decision_5m): от HOLD к BUY/STRONG_BUY/SELL

Исходное значение: `decision = "HOLD"`, список `reasons` заполняется по ходу.

### 3.1 Правила по 5m-данным

1. **RSI ≤ 32** и импульс 2ч ≥ −0.3% → `STRONG_BUY` («перепроданность, отскок»).
2. **RSI ≤ 38** и цена у 5-дневного минимума (≤ low_5d * 1.005) → `BUY`.
3. **RSI ≥ 76** → `SELL` («перекупленность»).
4. **RSI 68–76** → остаётся HOLD, в reasons добавляется «ближе к перекупленности, ждать».
5. **Импульс 2ч > 0.5%** и RSI < 62 → `BUY` («импульс за 2ч, RSI не перекуплен»).

Дополнительно:

- При волатильности 5m > 0.4% и уже BUY/STRONG_BUY — в reasons добавляется рекомендация лимитных ордеров; при > 0.6% и HOLD — «выжидать».

### 3.2 Коррекция по новостям KB

- **Очень негатив (sentiment < 0.35):** при текущем BUY/STRONG_BUY → `decision = "HOLD"`, «вход отложен», `kb_news_impact = "негатив (вход отложен)"`.
- **Негатив (sentiment < 0.4):** при STRONG_BUY → `decision = "BUY"`; при BUY — reasons + осторожность; `kb_news_impact` = «негатив (вход ослаблен)» / «негатив (осторожность)».
- **Позитив (sentiment > 0.65)** и текущий HOLD, RSI 38–52, импульс ≥ −0.5% → `decision = "BUY"`, «позитив в новостях поддерживает вход»; иначе при уже BUY/STRONG_BUY — только `kb_news_impact = "позитив"`.

### 3.3 Фаза сессии

- Если `session_phase in ("PRE_MARKET", "AFTER_HOURS", "WEEKEND", "HOLIDAY")` и decision уже BUY или STRONG_BUY → принудительно **`decision = "HOLD"`**, в reasoning добавляется «Вход отложен: сессия … — ждём открытия биржи 9:30 ET».

Итоговые `decision` и `reasoning` собираются в объект `d5` вместе со всеми входными и производными параметрами.

## 4. Крон: когда записывается BUY и что сохраняется

Файл: `scripts/send_sndk_signal_cron.py`, функция `process_ticker`.

### 4.1 Откуда берётся решение

- Один тикер: `d5 = get_decision_5m(ticker, use_llm_news=True)`.
- Кластерный запуск: `d5` берётся из `get_cluster_decisions_5m(..., use_llm_news=True)` (внутри тоже вызывается `get_decision_5m` по каждому тикеру).  
**Analyst LLM в этой цепочке не вызывается.**

### 4.2 Условия для рассылки и записи входа

Запись входа и отправка алерта выполняются только если одновременно:

1. `d5["decision"] in ("BUY", "STRONG_BUY")`;
2. по тикеру **нет открытой позиции** (GAME_5M или «любая»);
3. **регулярная сессия** (не PRE_MARKET/AFTER_HOURS/WEEKEND/HOLIDAY);
4. истёк **cooldown** после последнего сигнала по тикеру (`GAME_5M_COOLDOWN_MINUTES`);
5. `price` из `d5` есть и > 0.

Если что-то из этого не выполняется — рассылка и запись входа не делаются (логируются причины).

### 4.3 Запись в БД и контекст

При выполнении условий:

```python
entry_context = build_full_entry_context(d5)   # полный дамп параметров
record_entry(ticker, price, decision, reasoning, entry_context=entry_context)
```

- **decision** и **reasoning** — те, что вернул `get_decision_5m` (правила + KB + сессия).
- **entry_context** — снимок уровня «prompt_entry game_5m»: см. `services/deal_params_5m.py`, `build_full_entry_context(d5)`. В том числе:
  - `deal_params_version: 1`, `entry_impulse_pct` (= `momentum_2h_pct`), `decision`, `reasoning` (до 500 символов), `price`;
  - RSI, волатильность, session_high, period_str, стоп/тейк %, entry_advice, session_phase, kb_news_impact, бары, high_5d/low_5d, pullback, последние бары;
  - при наличии — premarket_gap_pct, minutes_until_open, prev_close, llm_insight, llm_sentiment (если был вызов LLM-новостей).

В `trade_history` в колонку `context_json` пишется сериализованный `entry_context`; при закрытии позиции в отчётах и в `/closed_impulse` импульс при входе берётся из этого же контекста (в т.ч. через `normalize_entry_context` для старых записей без полного дампа).

## 5. Где участвует LLM

### 5.1 LLM-новости (в get_decision_5m)

- **Включение:** `GAME_5M_USE_LLM_NEWS` и `USE_LLM_NEWS` в config.
- **Что делает:** добавляет в `d5` поля `llm_news_content`, `llm_sentiment`, `llm_insight` для отображения в алерте и в prompt_entry.
- **Влияние на decision:** **никакого** — решение BUY/HOLD/SELL считается только по правилам и KB.

### 5.2 Analyst LLM (get_decision_with_llm) — только отчёты и рекомендации

- **Где вызывается:** в Telegram-боте при запросах `/recommend5m`, `/prompt_entry game5m` и т.п., если в config/БД включён **USE_LLM** (для аналитика).
- **Вход:** тикер, при необходимости кластерный контекст (корреляция, сигналы по другим тикерам).
- **Выход:** отдельная рекомендация с учётом техданных и корреляции (может отличаться от правильного BUY/HOLD/SELL).
- **Использование:** только для **отображения** в виде «С учётом корреляции (LLM)» и для текста рекомендации пользователю. **Крон эту рекомендацию не использует** и в `record_entry` не подставляет; в БД при BUY попадает только результат `get_decision_5m`.

Итого: **решение, по которому реально открывается позиция и пишется в trade_history, формируется без Analyst LLM** — только правила 5m + новости KB + фаза сессии; LLM участвует в отображении и в дополнительной аналитике.

**Следствие:** вход в крон также делается **без учёта корреляций и кластера**.

## 7. Чем техническое решение отличается от LLM-рекомендации (в отчётах 5m)

В потоке prompt_entry game5m и /recommend5m при включённом USE_LLM вызывается не AnalystAgent, а **llm_service.analyze_trading_situation**: в него передаётся тот же технический результат (цена, RSI 5m, волатильность, **само правило-решение** как `technical_signal` и `strategy_signal`) и тот же новостной контекст из KB (`kb_news_summary` / `kb_news_impact`). Дополнительно в промпт попадает только **cluster_note** — текст с корреляциями и параметрами по другим тикерам кластера (цены, RSI, коэффициенты корреляции).

То есть **техническое решение для входа и LLM-рекомендация отличаются по сути только**:
1. **Дополнительным анализом корреляционных параметров** — LLM получает cluster_note (корреляции и контекст по связанным тикерам) и может скорректировать или обосновать рекомендацию с учётом кластера.
2. **Опционально — свежими новостями от LLM** (сейчас по умолчанию отключены: GAME_5M_USE_LLM_NEWS=false). Когда они включены, в get_decision_5m в d5 добавляются llm_news_content, llm_sentiment, llm_insight; они показываются в отчёте, но в вызов analyze_trading_situation в текущей реализации не передаются — туда идут только новости из KB. То есть «свежие LLM-новости» влияют на отображение и контекст в d5, но не на текст промпта к analyze_trading_situation.

Итого: да, верно — **отличие только в дополнительном анализе корреляций (и при желании в будущем — в подмешивании свежих LLM-новостей в промпт)**; база (5m-правила + KB-новости) одна и та же.

## 6. Краткая последовательность для одного BUY в кроне

1. Загрузка 5m OHLC по тикеру (Yahoo).
2. Расчёт RSI, волатильности, импульса 2ч, session_high, экстремумов последних баров.
3. Загрузка новостей KB за период, определение very_negative / recent_negative / recent_positive.
4. (Опционально) Запрос LLM-новостей по тикеру → поля в d5 для вывода.
5. Применение правил по RSI и импульсу → предварительный decision.
6. Коррекция по KB (негатив/позитив) → итоговый decision и reasoning.
7. Учёт сессии: вне REGULAR → принудительно HOLD.
8. Расчёт стоп/тейк из конфига и импульса, формирование полного `d5`.
9. В кроне: если decision in (BUY, STRONG_BUY), нет позиции, сессия регулярная, cooldown пройден → `build_full_entry_context(d5)` → `record_entry(..., entry_context=...)` → запись BUY в `trade_history` с полным снимком параметров в `context_json`.

Все параметры, которые видит пользователь в «prompt_entry game_5m», при новом входе сохраняются в этом снимке и затем могут использоваться для аналитики и для подмешивания в будущий контекст (в т.ч. через `normalize_entry_context` для старых записей с упрощённым форматом).

## 8. Закрытие позиции: причины выхода и запись цены

Крон при каждом запуске проверяет открытые позиции через `game_5m.should_close_position()`. Причина выхода пишется в `trade_history.signal_type` (в отчёте — колонка «Причина выхода»). Порядок проверок в коде фиксирован.

### 8.1 Причины выхода (signal_type)

| Причина | Когда срабатывает |
|--------|---------------------|
| **TAKE_PROFIT** | Цена (или макс. High последних баров) достигла уровня тейка. Уровень тейка считается от импульса 2ч при входе (или из конфига). |
| **STOP_LOSS** | Стоп включён (GAME_5M_STOP_LOSS_ENABLED) и цена (или мин. Low последних баров) достигла уровня стопа. |
| **TIME_EXIT** | (1) Последние N минут сессии (GAME_5M_SESSION_END_EXIT_MINUTES, по умолчанию 30) и текущий PnL ≥ GAME_5M_SESSION_END_MIN_PROFIT_PCT (напр. 0.3%); исключение: при сигнале STRONG_BUY не выходим. (2) **Либо** позиция старше GAME_5M_MAX_POSITION_DAYS (ограничение по сроку игры). |
| **SELL** | Правила 5m вернули решение **SELL** (напр. RSI(5m) ≥ 76 — перекупленность); крон по этому сигналу закрывает позицию. |

**Важно:** **SELL** — это не ручная продажа в Telegram. Ручного закрытия через бота в текущей схеме нет; все закрытия выполняет крон. SELL = автоматическое закрытие по сигналу «продать» от get_decision_5m (правила или LLM при стратегии llm).

**Ограничение по сроку игры** даёт **TIME_EXIT**, а не SELL. SELL — только реакция на решение правил «продать» (перекупленность и т.п.).

### 8.2 При выключенном стоп-лоссе (GAME_5M_STOP_LOSS_ENABLED=false)

Закрытие возможно только по:

- **TAKE_PROFIT** — достигнут уровень тейка;
- **TIME_EXIT** — конец сессии с минимальным профитом **или** истёк макс. срок в позиции (GAME_5M_MAX_POSITION_DAYS);
- **SELL** — правила вернули SELL (напр. RSI перекупленность).

### 8.3 TAKE_PROFIT: решение и запись цены по high

- **Решение «тейк достигнут»:** в `should_close_position` используется **price_for_take = max(current_price, bar_high)** (bar_high = макс. High последних 5m баров). Если pnl_take_pct ≥ порог тейка → возвращается TAKE_PROFIT.
- **Запись цены выхода в БД:** при закрытии по TAKE_PROFIT крон пишет **exit_price = bar_high** — ту же цену (максимум баров), по которой принято решение. Так цена закрытия в отчёте соответствует принятию решения по high.

Раньше в БД записывали **exit_bar_close** (close бара). Если бар давал пик вверх (~6%), но закрывался у открытия (+0.06%), в отчёт попадал TAKE_PROFIT с малым P/L %. После исправления при TAKE_PROFIT пишется bar_high. Код: `scripts/send_sndk_signal_cron.py` (блок при exit_type == "TAKE_PROFIT"); причина выхода и цена закрытия хранятся в `trade_history.signal_type` и в цене закрытия (записываемой при вызове close_position).
