# Отчёт по сверке GAME_5M: тейк на 5m vs 30m и эмуляция 30m

Документ описывает **файл JSON**, который пишет скрипт `scripts/backtest_game5m_take_5m_vs_30m.py` при запуске с `--json-out`, и как получить из него **читаемые таблицы** (Markdown).

---

## 1. Как получить JSON и таблицу

На инстансе (в контейнере `lse`, после пересборки образа с актуальным кодом):

```bash
cd ~/lse
docker compose exec lse python scripts/backtest_game5m_take_5m_vs_30m.py \
  --days 7 --exchange US --full-30m-sim \
  --json-out /app/logs/game5m_take_5m_vs_30m.json
```

На локальной машине после `scp`:

```bash
cd /path/to/lse
python3 scripts/render_game5m_take_json_to_md.py logs/game5m_take_5m_vs_30m.json -o docs/local_game5m_take_report.md
# или только в терминал:
python3 scripts/render_game5m_take_json_to_md.py logs/game5m_take_5m_vs_30m.json
```

Скрипт рендера: `scripts/render_game5m_take_json_to_md.py`.

## 2. Структура JSON (корень)

| Поле | Смысл |
|------|--------|
| `generated_at` | Время формирования отчёта (UTC, ISO). |
| `days` | Глубина выборки `BUY` из `trade_history` (календарные дни назад). |
| `exchange` | Код биржи в таблицах `market_bars_5m` / `market_bars_30m` (обычно `US`). |
| `rows` | Массив строк: **один фактический вход** → два реплея выхода (5m и 30m) + фактический `SELL`, если есть. |
| `full_30m_strategy_sim` | Есть только при запуске с **`--full-30m-sim`**: объект `{ "SNDK": [ {...}, ... ], ... }` — сделки **полной** 30m-стратегии на том же календарном окне. |
| `full_30m_window_et` | Границы окна эмуляции 30m (ET, ISO), если был `--full-30m-sim`. |

---

## 3. Таблица по `rows` (сверка на одних и тех же BUY)

Каждая строка таблицы соответствует одному элементу `rows[]`.

| Логика колонок | Пояснение |
|----------------|-----------|
| **Идентификация** | `buy_id`, `ticker`, `entry_ts`, `entry_price` — как в БД. |
| **5m_*** | Реплей: по барам 5m после входа, когда впервые сработал бы `should_close_position` (тейк/стоп/TIME_EXIT и т.д.). `5m_log_ret` — \(\ln(\text{exit}/\text{entry})\). |
| **30m_*** | То же, но импульс для тейка и шаг по времени считаются по **30m** (`compute_30m_features`, 2 ч = 4 бара). |
| **diff** | `log_ret_5m − log_ret_30m`. Удобно смотреть знак: кто «дал» большую log-доходность на этом входе. |
| **actual_*** | Следующий по времени `SELL` из истории (если есть): факт выхода в проде. |

Дополнительно в JSON (не в таблице рендера по умолчанию): `bars_5m_loaded`, `bars_30m_loaded` — сколько баров подгрузилось для окна вокруг входа (диагностика пустых реплеев).

---

## 4. Таблица по `full_30m_strategy_sim` (отдельный тест)

Это **не** «те же даты входа», а проход по 30m в окне **последние `days` дней ET** с **собственными** сигналами входа (`decide_game5m_technical` на 30m-признаках) и выходами.

| Поле сделки | Смысл |
|-------------|--------|
| `entry_ts` / `entry_price` | Вход по закрытию бара 30m в RTH, когда сработал BUY/STRONG_BUY. |
| `entry_branch` / `entry_decision` | Ветка правила и тип сигнала. |
| `exit_ts`, `exit_signal`, `exit_detail` | Первый выход по правилам `game_5m.should_close_position`. |
| `log_ret` | Log-return сделки. |

Премаркет 1m и KB в этой эмуляции **не** подмешиваются (офлайн-режим).

---

## 5. Интерпретация

- **Раздел `rows`:** отвечает на вопрос: «Если бы мы закрывали **эту же** позицию по 5m-логике vs по 30m-логике тейка, **когда** и **с какой log-доходностью**?»
- **Раздел `full_30m_strategy_sim`:** отвечает: «Сколько раз и как стратегия **сама** вошла бы и вышла на 30m за неделю?» — сравнение с частотой/качеством фактических BUY из `rows` — отдельный срез.

При расхождении 5m vs 30m в `rows` смотрите `momentum_2h_pct` / `take_pct_effective` в обеих колонках: на 30m импульс и порог тейка обновляются реже, поэтому момент выхода и цена часто отличаются.
