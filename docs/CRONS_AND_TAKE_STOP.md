# Два крона: 5m и портфель. Тейк/стоп и логи

## 1. Два скрипта — две «игры»

**Приоритет: прогнозы для вступления в игру.** Система даёт сигналы и параметры входа; играет трейдер. Ему нужна информация **по возможности раньше** — крон 5m каждые 5 мин, при необходимости премаркет-сводка с прогнозом на вступление (см. premarket_cron, PREMARKET_ENTRY_PREVIEW_5M).

| Скрипт | Назначение | Тикеры из config |
|--------|------------|------------------|
| **send_sndk_signal_cron.py** | Игра **5m**: интрадей по 5‑минутным данным, **прогноз для вступления** (BUY/STRONG_BUY) → рассылка в Telegram; выход по тейку/стопу или SELL. Трейдер принимает решение о сделке сам. | **GAME_5M_TICKERS** (если задан), иначе **TICKERS_FAST**. Пример: SNDK, MU. |
| **trading_cycle_cron.py** | **Портфельная игра**: дневные стратегии (Neutral, Momentum и т.д.), AnalystAgent + LLM, решение 2–3 раза в день. | **TRADING_CYCLE_TICKERS** (если задан), иначе **TICKERS_MEDIUM** + **TICKERS_LONG**. |

**Соответствие с командами бота:** крон 5m использует ту же логику, что `/prompt_entry game5m` и `/recommend5m` (прогноз для вступления). Крон портфеля — ту же, что `/prompt_entry portfolio`.

- **send_sndk_signal_cron** — только про 5m (частый поллинг, тейк/стоп от импульса 2ч и конфига GAME_5M_*).
- **trading_cycle_cron** — только про портфель (редкие запуски, тейк/стоп из стратегии или PORTFOLIO_TAKE_PROFIT_PCT / стоп по лог-доходности).

Оба пишут сделки в одну таблицу **trade_history**, но с разным **strategy_name**: `GAME_5M` у 5m, у портфеля — название стратегии (Neutral, Momentum и т.д.).

## 2. Могут ли тикеры пересекаться?

**Да.** Один и тот же тикер может быть и в GAME_5M_TICKERS, и в TRADING_CYCLE_TICKERS (например SNDK в обоих списках).

Важно не «какой скрипт обрабатывает тикер», а **кто открыл позицию**:

- Если **вход в позицию** сделал **send_sndk_signal_cron** (BUY с `strategy_name=GAME_5M`) — тейк/стоп по этой позиции проверяет **только send_sndk_signal_cron** (каждые 5 мин). Портфельный крон её не трогает.
- Если **вход** сделал **trading_cycle_cron** (ExecutionAgent, BUY с другой стратегией) — тейк/стоп проверяет **только trading_cycle_cron** (при своих запусках 9/13/17). Крон 5m для таких позиций только «видит» их в get_open_position_any (чтобы не слать повторный сигнал на вход), но закрытие по тейку/стопу делает портфельный агент.

Итого: пересечение тикеров допустимо; **владелец позиции** = тот, кто записал BUY (по полю strategy_name в trade_history).

## 3. Где проверяются тейк и стоп

### 3.1. Позиции GAME_5M (strategy_name = 'GAME_5M')

**Кто проверяет:** только **send_sndk_signal_cron.py**.

**Как часто:** при каждом запуске крона (например каждые 5 мин).

**Логика:** `services/game_5m.py` — `should_close_position()` (тейк с допуском 0.05%, стоп, SELL, TIME_EXIT). Цена для проверки: max(5m close, quotes).

**Что видно в логах (send_sndk_signal_cron):**

- В начале по каждому тикеру:
  - `[5m] SNDK: открытая_позиция=нет, цена_5m=596.39, решение=HOLD`
  - `[5m] MU: открытая_позиция=да [GAME_5M], цена_5m=406.90, решение=HOLD`
- Итог по тикеру:
  - `[5m] MU: итог — позиция открыта: вход=395.83, текущая=406.90, pnl=2.80%, тейк=2.7%, стоп=... — тейк/стоп не сработали; решение HOLD → нет сигнала на вход`
  - или `[5m] MU: итог — позиция закрыта по TAKE_PROFIT @ 406.50`

Тейк/стоп по 5m — **только в логах send_sndk_signal_cron**, не в trading_cycle_cron.

---

### 3.2. Позиции портфеля (любая стратегия, кроме GAME_5M)

**Кто проверяет:** только **trading_cycle_cron.py** через **ExecutionAgent.check_stop_losses()**.

**Как часто:** при каждом запуске trading_cycle_cron (например 9:00, 13:00, 17:00 пн–пт), **после** цикла по тикерам (анализ → BUY/SELL/HOLD).

**Логика:** `execution_agent.py` — для каждой открытой позиции из portfolio_state (с ценой входа из trade_history):

1. Стоп: лог-доходность ≤ порога (~ −5%) → закрытие по стопу.
2. Если стратегия позиции = GAME_5M → **пропуск** (тейк/стоп по ней проверяет send_sndk_signal_cron).
3. Тейк: берётся `take_profit` из последней BUY по тикеру или **PORTFOLIO_TAKE_PROFIT_PCT**; если pnl% ≥ тейка → закрытие по тейку.

**Что видно в логах (trading_cycle_cron):**

- Один раз в начале проверки:
  - `Проверка стоп‑лоссов по открытым позициям`
  - или `Открытых позиций нет, стоп‑лоссы не проверяются`
- Для каждой открытой позиции (кроме GAME_5M):
  - `Проверка стоп‑лосса для MSFT: entry=..., current=..., log_ret=..., threshold=...`
  - затем либо:
    - `Тейк‑профит для MSFT: порог=3.0%, pnl=2.50% — не достигнут`
    - или `Тейк‑профит для MSFT не задан (задайте PORTFOLIO_TAKE_PROFIT_PCT в config.env, напр. 3.0)`
    - или закрытие: `Take-profit triggered: pnl=...% >= ...%`
  - и `Стоп‑лосс для MSFT не сработал (log_ret=... > ...)`
- Для позиций GAME_5M в этом же запуске:
  - `MSFT — GAME_5M, тейк/стоп проверяет send_sndk_signal_cron` (если MSFT вдруг был открыт как GAME_5M)

Тейк/стоп по портфелю — **только в логах trading_cycle_cron**, в блоке после анализа тикеров.

| Что ищешь | Где смотреть |
|-----------|----------------|
| Есть ли открытая позиция по 5m, тейк/стоп по 5m | Логи **send_sndk_signal_cron**: строки `[5m] TICKER: открытая_позиция=...` и `[5m] TICKER: итог — ...` |
| Тейк/стоп по портфелю (Neutral, Momentum и т.д.) | Логи **trading_cycle_cron**: блок «Проверка стоп‑лоссов», строки «Проверка стоп‑лосса для TICKER», «Тейк‑профит для TICKER» |

| Решение на вход по портфелю | Логи **trading_cycle_cron**: «Сигнал AnalystAgent (с LLM) для TICKER: HOLD/BUY», «Сигнал HOLD для TICKER, покупка не выполняется» |

## 5. Конфиг: тикеры и тейк/стоп

### 5.1 Тейк по импульсу (5m)

**Формула:** если **импульс_2ч** (реализованное движение цены за последние ~2 ч) **≥ GAME_5M_TAKE_PROFIT_MIN_PCT** (4%):  
`тейк = min(импульс_2ч × GAME_5M_TAKE_MOMENTUM_FACTOR, потолок)`.  
Иначе тейк = **потолок** (GAME_5M_TAKE_PROFIT_PCT или GAME_5M_TAKE_PROFIT_PCT_&lt;TICKER&gt;).

Поэтому при импульсе 6% тейк срабатывает на **6%**, а не на 7%: потолок 7% только ограничивает сверху; фактический тейк берётся от импульса. Потолок по тикеру: **GAME_5M_TAKE_PROFIT_PCT_SNDK=8** — для SNDK макс. тейк 8%; для остальных — GAME_5M_TAKE_PROFIT_PCT.

### 5.2 Параметры

- **5m:** тикеры — **GAME_5M_TICKERS** или TICKERS_FAST; тейк/стоп — **GAME_5M_TAKE_PROFIT_PCT** (потолок), **GAME_5M_TAKE_PROFIT_MIN_PCT** (с какого импульса считать тейк от импульса), **GAME_5M_TAKE_MOMENTUM_FACTOR** (ниже 1 — консервативнее, выше 1 — агрессивнее до потолка; в коде clamp примерно 0.3–2.0), **GAME_5M_TAKE_PROFIT_PCT_SNDK** и т.д. (потолок по тикеру); **GAME_5M_STOP_LOSS_PCT**, **GAME_5M_STOP_LOSS_ENABLED** и др.
- **Портфель:** тикеры — **TRADING_CYCLE_TICKERS** или TICKERS_MEDIUM + TICKERS_LONG; тейк по умолчанию — **PORTFOLIO_TAKE_PROFIT_PCT** (если 0 или не задан — в логах «Тейк‑профит не задан»). Стоп-лосс: **PORTFOLIO_STOP_LOSS_ENABLED** (true/false); при false в логах предупреждение «стоп отключён», закрытие только по тейку. Порог стопа — **STOP_LOSS_LEVEL** (0.95 ≈ −5%). Оба параметра (PORTFOLIO_STOP_LOSS_ENABLED, STOP_LOSS_LEVEL) можно переопределить в БД (strategy_parameters, GLOBAL) — тогда trading_cycle_cron и /gameparams используют значение из БД.

См. также: [TICKER_GROUPS.md](TICKER_GROUPS.md), [CRON_TICKERS_EXPLANATION.md](CRON_TICKERS_EXPLANATION.md).

## 6. Проверка: сигнал 5m пришёл, но позиция не в /pending

Цепочка по шагам:

1. **Крон** `send_sndk_signal_cron.py` при решении BUY/STRONG_BUY вызывает `record_entry(ticker, price, ...)` из `services/game_5m.py`.
2. **record_entry** делает `INSERT INTO public.trade_history (..., side='BUY', strategy_name='GAME_5M', ...)`. Подключение к БД: `create_engine(get_database_url())`; конфиг берётся из `config_loader.load_config()` → файл `Path(config_loader.__file__).parent / "config.env"` (рядом с config_loader.py в корне проекта).
3. **Бот** по команде /pending вызывает `report_generator.get_engine()` → `create_engine(get_database_url())`, затем `load_trade_history(engine)` без фильтра по стратегии → `SELECT * FROM trade_history ORDER BY ts, id`, затем `compute_open_positions(trades)` → считает открытые позиции (BUY без полного SELL по тикеру).

Чтобы позиция из шага 1 попала в шаг 3, **крон и бот должны подключаться к одной и той же БД**:

- Один и тот же **config.env**: оба процесса загружают конфиг из пути относительно `config_loader.__file__` (каталог, где лежит config_loader.py). Если крон запускается из другого каталога или другого клона репозитория — может подхватиться другой config.env с другим DATABASE_URL.
- Переменные окружения: `get_config_value()` сначала смотрит `os.getenv(key)`. Если в crontab задан `DATABASE_URL=...` или бот запущен с `export DATABASE_URL=...`, а второй процесс — без него, получится разная БД.

**Что проверить:** в логах крона после записи входа есть строка вида `game_5m: вход SNDK id=123 @ 567.70 qty=... BUY` и (при следующем изменении) `[5m] БД: host=... database=lse_trading`. Убедитесь, что бот при /pending подключается к тому же host и database (например, добавьте временно лог в боте при старте или в обработчике /pending: какой DATABASE_URL используется, с замаскированным паролем).

## 7. Нереальная цена выхода в БД (системный сбой)

Если на графике 5m маркер «Тейк (прибыль)» висит **выше реального хая сессии** (например тейк @ 400 при хае 391) — в **trade_history** при закрытии попала завышенная цена. Откуда она могла взяться:

| Источник | Описание |
|----------|----------|
| **Крон 5m** (`send_sndk_signal_cron`) | Цена закрытия = `max(price_5m, price_quotes)`, затем ограничивается `bar_high` (тейк) / `bar_low` (стоп). Если **bar_high** не пришёл из `get_decision_5m` (или был 0) или в 5m данных был **битый бар** (High/Close = 400), в БД могла записаться нереальная цифра. |
| **Quotes** | `get_latest_prices()` / `quotes.close` — последняя строка по тикеру. Ошибка даты (другая сессия), неверный тикер или баг загрузки дают неправильный close. |
| **Ручное закрытие** | `scripts/close_game_position.py` берёт цену из `quotes` (последний close). Та же риск — неверная или устаревшая строка в quotes. |

**Что сделано для защиты:**

1. **game_5m.close_position()**  
   - При **TAKE_PROFIT**: записываемая цена не выше `bar_high` (если передан) и не выше **entry × 1.15** (если bar_high не передан).  
   - При **STOP_LOSS**: не ниже `bar_low` или **entry × 0.85**.  
   - При отклонении цены от входа больше чем на ±25% пишется **WARNING** в лог.

2. **Крон** перед вызовом `close_position` ограничивает цену по `bar_high`/`bar_low` и логирует:  
   `[5m] TICKER закрытие: тип=..., price_5m=..., price_quotes=..., bar_high=..., bar_low=... → exit_price=...`  
   По этому логу можно увидеть, откуда пришла завышенная/заниженная цифра.

3. **close_game_position.py**  
   Перед закрытием выводит предупреждение, если цена из quotes отличается от входа больше чем на 20%; `close_position` при этом ограничит запись в БД (entry ±15%).

4. **График 5m** (Telegram /chart5m)  
   Координаты маркеров тейка/стопа **обрезаются** диапазоном цен сессии (Low–High), чтобы даже при ошибочной записи в БД маркер не рисовался «в небе».

**Как разобрать уже попавшую ошибку:** в логах крона за момент закрытия искать строку `[5m] MU закрытие: ...` и `game_5m: MU закрыта @ ...`. Если в логе уже видно `exit_price=400` при `bar_high=391` — до правок ограничение не срабатывало (bar_high мог быть None из‑за отсутствия 5m или битого ответа). После правок такая запись не должна появляться; при сомнительной цене в лог пишется WARNING и цена ограничивается.

## 8. Кластерный контекст: как вызывается и как попадает в промпт (контекст по всем бумагам, рекомендации по отдельности)

**Идея:** один общий контекст по всему кластеру (все бумаги портфеля + матрица корреляций), а рекомендация запрашивается у LLM **отдельно по каждому тикеру**. В промпт по тикеру подставляется строка корреляций именно этого тикера с остальными и (в кроне) сигналы по уже обработанным тикерам в этом запуске.

**Где формируется кластер:**

| Место | Список для корреляций | По ком рекомендация |
|-------|------------------------|---------------------|
| **trading_cycle_cron** | Полный список портфеля (MEDIUM + LONG, включая индикаторы ^VIX и т.д.) — `cluster_tickers` | По каждому тикеру из списка *торгуемых* (без индикаторов), по одному вызову LLM на тикер |
| **Telegram /prompt_entry &lt;ticker&gt;** | Тот же полный список портфеля (`get_tickers_for_portfolio_game()`); если тикера нет в списке (например только в 5m), он добавляется в кластер | Один тикер — один вызов LLM |

**Как это работает по шагам:**

1. **Один раз** строится список кластера (все бумаги) и по нему считается матрица корреляций за 30 дней.
2. Для **каждого** тикера, по которому нужна рекомендация:
   - в промпт добавляется блок: «Кластер тикеров: MSFT, MU, SNDK, … Ты анализируешь один из них»;
   - в этот же блок подставляется **строка корреляций текущего тикера** с остальными (например для MSFT: «Корреляция этого тикера с другими: MU +0.85, SNDK +0.72, …»);
   - в **trading_cycle_cron** дополнительно подставляются **сигналы по другим тикерам в этом запуске** (кто уже обработан — BUY/HOLD и т.д.).
3. LLM получает один общий контекст по кластеру и одну «роль»: «анализируешь тикер X»; ответ (BUY/HOLD/STRONG_BUY) — одна рекомендация по этому тикеру.

Итого: **контекст один на весь кластер (все бумаги), рекомендации считаются по отдельности по каждому тикеру.** Для MSFT в промпт подтягивается корреляция с MU (и с остальными), для MU — корреляция с MSFT (и с остальными), матрица одна и та же, в промпте показывается только строка для текущего тикера.

**Код:** формирование контекста — `execution_agent.run_for_tickers(..., cluster_tickers=...)` и в боте `_handle_prompt_entry`; добавление блока в промпт — `analyst_agent.get_decision_with_llm(ticker, cluster_context=...)` → `technical_data["cluster_note"]` → LLM.

**Структура по играм в Telegram:** `/prompt_entry` — шаблон принятия решения по игре. Аргумент: игра (portfolio, game5m) или тикер — контекст строится по параметрам и тикерам этой игры. Выгрузка: отчёт для человека; для портфеля в отчёте также промпт для LLM (system, user, ответ). `portfolio`/`all` — портфельная игра (кластер MEDIUM+LONG, AnalystAgent); `game5m` — игра 5m (кластер GAME_5M_TICKERS, решение по правилам). Промпт для LLM есть только в портфельной игре.

## 9. Кластерный контекст в истории сделок (портфель)

При входе в сделку (BUY) портфельный крон записывает в **context_json** снимок кластера — что именно видел LLM при принятии решения. Так по истории сделок можно объективно оценить влияние корреляции и сигналов по другим тикерам на исход.

**Структура `context_json.cluster` (только для BUY от trading_cycle_cron с кластером, см. раздел 8):**

| Поле | Описание |
|------|----------|
| **tickers** | Список тикеров кластера в этом запуске. |
| **correlation_this_ticker** | Корреляция этого тикера с остальными (30 дн.): `{"MU": 0.85, "SNDK": 0.72}`. |
| **other_signals_at_decision** | Сигналы по другим тикерам на момент решения: `{"MU": "BUY", "SNDK": "HOLD"}` (те, что уже были обработаны в этом цикле). |

**Пример анализа по истории:**

- Сопоставить закрытые пары BUY→SELL по одному тикеру, вычислить PnL/win rate.
- Разбить сделки на группы: «в other_signals_at_decision был ещё хотя бы один BUY по коррелированному тикеру» vs «все остальные — HOLD/SELL».
- Сравнить средний PnL или долю прибыльных сделок между группами — даст объективную картину, помог ли контекст по коррелированным парам или нет.

Интерпретация величины корреляции и новостей по связанным тикерам заранее неочевидна; накопленная история сделок покажет, как кластерный контекст реально повлиял на решения LLM.
