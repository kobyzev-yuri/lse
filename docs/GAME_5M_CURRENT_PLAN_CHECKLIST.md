## Что нужно для работы «как есть»

1. **Конфиг**  
   В `config.env` (или переменные окружения) задать минимум:
   - `DATABASE_URL` — для trade_history и quotes.
   - Параметры 5m по желанию: `GAME_5M_TAKE_PROFIT_PCT`, `GAME_5M_STOP_LOSS_PCT`, `GAME_5M_STOP_LOSS_ENABLED`, `GAME_5M_ENTRY_STRATEGY` (= `technical` или `llm`).  
   Для LLM-входа дополнительно: `OPENAI_API_KEY` (или proxy), список тикеров и при необходимости `GAME_5M_CORRELATION_CONTEXT`.

2. **Крон**  
   Запускать `scripts/send_sndk_signal_cron.py` по расписанию (например каждые 5 минут в сессию), чтобы генерировались сигналы, открывались/закрывались позиции и писались сделки в БД с полным context_json.

3. **Данные**  
   - 5m-свечи приходят с yfinance при каждом вызове (в БД не обязательны для текущего плана).  
   - Для KB-новостей и сессии: таблицы `knowledge_base`, контекст сессии (`market_session`); при использовании suggested params — скрипт `daily_5m_params.py` и (опционально) `USE_SUGGESTED_5M_PARAMS=true`.

4. **Проверка**  
   - Один тикер: вызвать `get_decision_5m(ticker)` или через Telegram/веб (например `/prompt_entry game5m` по тикеру).  
   - Убедиться, что в закрытых сделках в context_json есть нужные поля (в т.ч. atr_5m_pct, volume_vs_avg_pct при наличии данных).

Этого достаточно, чтобы текущий план (правила + LLM, без CatBoost) работал в проде.

---

## Опционально

- **Расширение признаков** — добавлять новые поля в `compute_5m_features`; при необходимости дописать ключи в `deal_params_5m.FULL_ENTRY_KEYS`, чтобы они попадали в context_json.
- **Тесты** — юнит-тест на `compute_5m_features` (мок df 5m) и на то, что `get_decision_5m` возвращает структуру с полями из features.
- **Мониторинг** — логи крона, алерты при ошибках, периодический просмотр `/closed` и `/closed_impulse` для оценки качества решений.

## Позже (когда будете делать разметку по исходам)

- **Сбор срезов** — при каждом входе уже пишется context_json (признаки на момент входа); при закрытии — signal_type и цена. Этого достаточно, чтобы потом строить датасет «признаки при входе → исход (тейк/стоп/время/профит)».
- **Правила разметки** — определить целевые метки (например: хороший вход = TAKE_PROFIT или P/L% > X; плохой = STOP_LOSS или P/L% < −Y).
- **Пайплайн обучения** — скрипт: выборка из trade_history + парсинг context_json → признаки + метка; обучение CatBoost/другой модели; сохранение артефакта и опция `GAME_5M_ENTRY_STRATEGY=ml` с вызовом модели вместо правил/LLM.

Отдельный документ под формулировку правил разметки и формат датасета можно добавить, когда приступите к этому этапу.
