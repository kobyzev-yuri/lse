# CatBoost для игры 5m: план обучения и эксплуатации

Модель **не переписывает** поле `decision` (чистые правила). Дополнительно: `catboost_entry_proba_good`, статус, и при необходимости **`technical_decision_effective`** для входа/LLM (см. `docs/GAME_5M_CATBOOST_FUSION.md`, `services/catboost_5m_signal.py`).

---

## Что именно предсказывает CatBoost

- **Тип задачи:** бинарная классификация (логистическая потеря). На выходе — **оценка вероятности** `P(y=1 | X)`, где **X** — признаки на **момент входа** (как в сохранённом `context_json` + тот же набор полей, что в инференсе).
- **Что такое y=1 (цель обучения):** по умолчанию в `train_game5m_catboost.py` это **«закрытая сделка с положительным `net_pnl`»** (после комиссий в вашей модели PnL). Опция `--label log_return_pos` задаёт y=1 при **положительной лог-доходности** реализованной сделки (согласовано с принятой в проекте метрикой движения цены).
- **Чего модель не делает:** не предсказывает цену, не выбирает размер позиции, не заменяет стоп/тейк и не оптимизирует момент выхода. Она лишь говорит: «по похожим **входным** паттернам в прошлом чаще/реже получался исход с y=1».
- **Интерпретация для трейдера:** поле `catboost_entry_proba_good` — **справочный скор**, а не гарантия. Вероятности могут быть плохо откалиброваны при малой выборке.

### Как не «повторять ошибки» постфактум-анализа

Анализ **уже совершённых** сделок находит причины убытков (режим рынка, новости, корреляции, слишком поздний вход и т.д.). **Сама модель ошибки не исправляет:** она учится на факте исхода (y). Чтобы снизить риск воспроизвести те же проблемы:

1. **Не подменять правила входа скором** (пока нет A/B): технические правила остаются источником решения; CatBoost — второе мнение, иначе вы смешаете старые ошибки политики с новым сигналом.
2. **Честные признаки на входе:** при обучении корреляция берётся **только из `context_json`** на момент сделки, а не из «текущей» матрицы — иначе модель видит информацию, которой не было при входе (**утечка**).
3. **Валидация по времени** (как в скрипте): меньше самообмана, чем при случайном split; падающий AUC на свежем хвосте — сигнал к переобучению или смене режима.
4. **Порог объёма (`GAME_5M_CATBOOST_MIN_TRAIN_ROWS`):** при малом N модель не сохраняется — меньше шанс «зазубрить» шум из разбора убыточных сделок.
5. **Переобучение после смены логики игры** (`GAME_5M_RULE_VERSION`, новые поля в дампе): иначе модель продолжит оптимизировать старый мир, где ошибки были другими.
6. **Учёт смещения выборки:** в истории только сделки, которые **реально открылись** по крону/правилам; модель не знает контрфакт («что было бы на всех HOLD-барах»). Выводы аналитики про «лучше не входить в таких условиях» полезно позже кодировать **явными признаками или правилами**, а не ожидать, что CatBoost сам выведет запрет из пары десятков примеров.
7. **Мониторинг на новых закрытиях:** сравнивать долю y=1 при высоком/низком скоре на **новых** сделках; при деградации — отключить CatBoost или переобучить.

## 1. Данные

| Источник | Что берём |
|----------|-----------|
| `trade_history` | `strategy_name = 'GAME_5M'`, хронологический порядок |
| Закрытые сделки | `report_generator.compute_closed_trade_pnls(load_trade_history(...))` |
| Признаки **X** | `context_json` с **BUY** (вход) — полный дамп `build_full_entry_context` |
| Целевая переменная **y** | По умолчанию: `1` если `net_pnl > 0`, иначе `0` (бинарная «прибыльный вход») |

**Лог-доходность:** в проекте PnL и риск считаются через log-returns; для метки можно позже добавить режим `y = 1` при `log_return > 0` (эквивалентно росту цены выхода относительно входа при той же методике, что в `TradePnL`).

**Корреляция (`cb_corr_*`):** в JSON попадает при записи входа кроном (матрица на момент прогона). При обучении **не** вызывается `load_game5m_llm_correlation` для строки — только значения из `context_json`; отсутствующие → `0` (старые сделки).

---

## 2. Объём выборки и качество

- Минимум строк задаётся **`GAME_5M_CATBOOST_MIN_TRAIN_ROWS`** (например `50`–`100`). Меньше — скрипт завершается без записи `.cbm`, в проде остаётся `no_model_file` / прежняя модель.
- Классы часто несбалансированы: смотреть долю `y=1`; при необходимости `class_weights` / `scale_pos_weight` в CatBoost (в скрипте — параметр или правка после первых прогонов).
- **Валидация по времени:** последние **N%** закрытых сделок (по времени выхода) — hold-out, без перемешивания (меньше оптимистичной утечки).

---

## 3. Признаки и совместимость

- Схема колонок: `get_catboost_feature_schema()` в `catboost_5m_signal.py` = `["ticker"] + NUMERIC_FEATURE_KEYS`, категориальная только **`ticker`** (индекс `0`).
- Любое изменение списка признаков → **переобучить** модель и обновить **`game5m_entry_catboost.meta.json`**; иначе инференс вернёт `feature_mismatch`.

---

## 4. Обучение (локально / CI)

1. Установить CatBoost: `pip install -r requirements-catboost.txt` (или `pip install catboost`).
2. Убедиться, что в БД достаточно закрытых `GAME_5M` с непустым `context_json` на BUY.
3. Запуск:
   ```bash
   cd /path/to/lse && python scripts/train_game5m_catboost.py
   ```
   Опции: `--min-rows 80`, `--valid-ratio 0.2`, `--out local/models/game5m_entry_catboost.cbm`, `--dry-run` (только статистика).
4. Артефакты (не в git): `local/models/game5m_entry_catboost.cbm` + `game5m_entry_catboost.meta.json`.
5. В `config.env`: `GAME_5M_CATBOOST_ENABLED=true`, при необходимости `GAME_5M_CATBOOST_MODEL_PATH=...`.

---

## 5. Регулярное переобучение

| Вариант | Когда |
|---------|--------|
| Вручную | Раз в неделю/месяц после накопления закрытых сделок |
| Cron | Например раз в неделю вне рынка: тот же `train_game5m_catboost.py`, бэкап старого `.cbm` перед заменой |
| Порог | Переобучать при смене `GAME_5M_RULE_VERSION` или после добавления полей в признаки |

---

## 6. Ограничения и риски

- Мало данных → переобучение и нестабильная калибровка вероятностей; модель **только справочно**.
- Сдвиг режима рынка → старые веса устаревают; нужен мониторинг доли верных предсказаний на новых закрытиях (будущий «игрок B»).
- Выживаемость: в истории только реализованные входы по текущим правилам — смещение выборки относительно «всех баров».

## 8. Ежедневный конвейер и что смотреть при каждом обучении

Скрипт **`scripts/run_daily_game5m_ml_pipeline.py`** (cron после сессии) последовательно:

1. `build_game5m_stuck_dataset.py` → CSV + **`.meta.json`** (число строк, окно дат, доли классов / колонки — см. файл meta рядом с CSV).
2. `build_game5m_continuation_dataset.py` → CSV + **`.meta.json`**.
3. `train_game5m_catboost.py` (если не `DAILY_ML_SKIP_CATBOOST=1`) → `game5m_entry_catboost.cbm` + **`game5m_entry_catboost.meta.json`**.
4. Дописывает строку в **`game5m_daily_ml_report.jsonl`** (путь переопределяется `DAILY_ML_REPORT_JSONL`).

**Полнота датасетов (каждый прогон)**

| Артефакт | На что смотреть |
|----------|-----------------|
| `stuck_dataset.*.meta.json` | Число строк ≥ разумного минимума для обучения stuck-модели (когда появится отдельный train); резкий спад строк относительно числа закрытых GAME_5M — проверить пустой `context_json` на BUY или фильтры скрипта. |
| `continuation_dataset.*.meta.json` | То же для underprofit/continuation; мало `TAKE_PROFIT*` выходов → мало строк — нормально в «тихую» неделю. |
| `train_game5m_catboost` exit code **2** | Недостаточно строк (`DAILY_ML_MIN_CATBOOST_ROWS`, по умолчанию 35) — модель **не перезаписана**; в JSONL будет `note` об этом. |
| **Утечки** | При смене схемы признаков или `GAME_5M_RULE_VERSION` сравнивать `feature_mismatch` / переобучение на hold-out (в meta обычно `auc_valid`, `n_train`, `n_valid`). |

**Метрики CatBoost при каждом новом `.cbm`**

| Метрика | Где | Интерпретация |
|---------|-----|----------------|
| `auc_valid` | `game5m_entry_catboost.meta.json` и строка JSONL | Временной hold-out; падение на новых неделях при росте на train → переобучение или смена режима рынка. |
| `n_train` / `n_valid` | там же | Малый `n_valid` → шумная оценка AUC. |
| `trained_at` | там же | Версионирование «какая модель на проде». |
| **Эксплуатация** | `/api/analyzer` → `catboost_entry_backtest` | Доля скоров `ok`, калибровка P(win) vs loss по бакетам; много `disabled` → не включён CatBoost или нет файла модели на хосте веба. |

**Не путать:** nightly JSONL — про **подготовку данных и учёбу**; анализатор — про **факт сделок и правила** на выбранном окне. Оба должны улучшаться согласованно после смены логики входа/выхода.

## 9. Портфельная игра и «вилка» цен на горизонте (CatBoost как инструмент)

Сейчас CatBoost в репозитории завязан на **5m** и **бинарный** скор входа (`catboost_entry_proba_good`), а не на диапазон цен. Для **портфельной игры** (`trading_cycle`, дневные `quotes`, `StrategyManager`) тот же **CatBoost** можно использовать **отдельной моделью** как инструмент для **предсказания вилки** (например квантили или несколько регрессоров на горизонт H):

- **Цели (примеры):** лог-доходность от `close` на день +5 / +10 / +20; или верхняя/нижняя граница пути (max/min **daily** high/low в окне H от точки входа), обученные только на признаках, доступных **до** горизонта (без утечки будущих OHLC внутри окна).
- **Признаки:** дневные SMA/RSI/вола, VIX, KB-bias (как в карточке портфеля), при необходимости агрегаты корреляции из `context_json` на BUY — по тому же принципу честности, что в §1 для 5m.
- **Интеграция:** не подменять сразу `PORTFOLIO_TAKE_PROFIT_PCT` — сначала **обоснование** в отчёте анализатора / карточке («модельная вилка ±…% на H дней»), затем опционально порог на вход или подсказка к цели тейка после A/B.

Это сосуществует с ML Керима и LLM: CatBoost даёт **числовой слой** по истории, LLM — **квалификация** контекста, правила — **исполняемая политика**.
