# План перехода на кластерные стратегии и ИИ-рефлексию (RLM)

Данный документ описывает поэтапный план перехода системы алгоритмической торговли на использование динамических параметров, машинного обучения на основе исторического опыта сделок (RLM - Reinforcement Learning from Market) и авто-кластеризации активов. Главный принцип: **вносить изменения безопасно, не ломая текущую логику бота (backwards compatibility).**

## ЭТАП 1: Настройка телеметрии и динамических параметров (База для обучения)
*Текущий фокус. Формируем базис для сбора данных, на которых LLM будет учиться, ничего не ломая в принятии решений.*

- [ ] **1.1. Обновление схемы БД (Телеметрия сделок)** 
  - Добавить в `trade_history` новые столбцы для сбора полного контекста: `take_profit`, `stop_loss`, `mfe` (Max Favorable Excursion - макс. плавающая прибыль), `mae` (Max Adverse Excursion - макс. плавающий убыток), `context_json` (запись состояния индикаторов, гэпа, sentiment и др.).
- [ ] **1.2. Создание таблицы `strategy_parameters`**
  - Создать таблицу в PostgreSQL для хранения настроек (пороги входа, чувствительность) вместо жесткого кода, с возможностью привязки к тикеру или кластеру типа `MEMORY_CHIPS`.
- [ ] **1.3. Адаптация ExecutionAgent и Стратегий**
  - Научить код писать расширенные данные в БД при открытии и закрытии сделки.
  - Научить `StrategyManager` читать параметры из БД при наличии (с fallback на дефолты в коде, чтобы ничего не упало).
- [ ] **1.4. Простой UI настроек**
  - Добавить страницу в веб-интерфейсе для ручного просмотра и изменения порогов стратегий «на лету».

## ЭТАП 2: Подключение LLM Рефлексии (Continuous Learning)
*Включаем AI-анализ закрытых сделок для генерации уроков.*

- [ ] **2.1. Скрипт разбора сделок (`evaluate_trades_cron.py`)** 
  - Раз в день система берет закрытые сделки и передает их в LLM вместе с контекстом сделки (`context_json`) и финансовым результатом (PnL, причина закрытия).
- [ ] **2.2. Промпты для рефлексии** 
  - LLM оценивает, почему стратегия сработала или не сработала по конкретному кластеру/тикеру, и вырабатывает четкий "Урок" или предложение по изменению параметра.
- [ ] **2.3. Векторная память ошибок и успехов** 
  - Сохранение полученных уроков в текущей таблице `knowledge_base` с типом `event_type='TRADING_LESSON'`.
- [ ] **2.4. Интеграция уроков в AnalystAgent** 
  - Перед принятием новых решений AnalystAgent делает легкий векторный поиск релевантных "Уроков" и передает их как дополнительный контекст (Guardrails).

## ЭТАП 3: Авто-Кластеризация (Dynamic Regime Discovery)
*Алгоритм перестает мыслить отдельными тикерами и начинает видеть временные связи (режимы) на рынке.*

- [ ] **3.1. Анализ скользящей корреляции**
  - Создать модуль расчета скользящей корреляции и коэффициента Beta (масштаба) для пар активов.
- [ ] **3.2. Динамическое группирование**
  - Алгоритм автоматически объединяет тикеры в кластеры при высокой корреляции в течение окна времени (например, 30 дней) и расформировывает их при падении корреляции.
- [ ] **3.3. Кластерные сигналы и Нормализация ATR** 
  - Торговые стратегии начинают учитывать кластер как единую сущность. 
  - Отказ от жестких процентов. Индивидуальные `take_profit` и `stop_loss` каждого тикера высчитываются через его ATR (Average True Range). Если у MU волатильность выше SNDK, стоп для MU будет шире.
- [ ] **3.4. Изоляция и защита в песочнице** 
  - Повсеместное тестирование новых режимов перед включением.
