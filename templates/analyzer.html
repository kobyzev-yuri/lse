Анализатор эффективности сделок

Тот же расчёт, что и в Telegram: /analyser. Период — 1–30 дней. Ниже краткая выжимка на русском; полный отчёт — в JSON с GET /api/analyzer (удобно сохранить или скормить скрипту).

Справка по метрикам (поля JSON)

Проценты по окну сделки считаются на 5m-барах от входа до выхода (Yahoo 5m, ET). Фактический результат — realized_pct (с комиссиями). Формулы и контекст: docs/GAME_5M_CALCULATIONS_AND_REPORTING.md в репозитории.

Фактический результат (realized_pct)
Доходность сделки в % к cost-basis; в развёрнутом JSON также log-return.
Недобор до максимума окна (missed_upside_pct)
Сколько не дожали до лучшей цены по окну: max(0, potential_best_pct − realized_pct) по high 5m. В сводке сумма недоборов — поле sum_missed_upside_pct. Сумма только по прибыльным — sum_missed_upside_pct_on_wins.
Сумма «избежимых просадок» по окну (sum_avoidable_loss_pct)
По сделке: max(0, realized_pct − preventable_worst_pct), где preventable_worst_pct — минимум по Low окна относительно входа. На прибыльных long сумма часто большая: вы зафиксировали плюс, а «худшая точка» окна была глубоко в минусе — это не «сумма убытков, которых можно было избежать».
late_polling_signals / exit_below_window_mfe_count
Число сделок, где цена выхода отстаёт от max High 5m-окна вход→выход примерно >0,4% и при этом missed_upside_pct > 0.3. Прокси «вышли заметно ниже пика окна» (часто нормально для лимитного тейка). Не интервал опроса cron и не задержка Telegram.
Худшие исходы (top_cases.top_losses)
Сортировка по возрастанию realized_pct. При высоком win rate здесь часто «самые слабые плюсы», а не убытки.
Крупнейший недобор — все сделки (top_cases.top_missed_upside)
Сортировка по убыванию missed_upside_pct; без отбора по знаку результата (и убыток, и прибыль).
Прибыльные с крупным недобором (top_cases.top_profitable_missed_upside)
Только realized_pct > 0, по убыванию missed_upside_pct: плюс по сделке, но заметный запас до high окна (типичный разбор тейка / потолка).

В summary также exit_below_window_mfe_count (= устаревшее имя late_polling_signals), high_vol_losses_count, … Полный список сделок — trade_effects при include_trade_details=1.

Скачать и открыть отчёт (JSON, GET /api/analyzer)

Один запрос возвращает весь отчёт: meta, summary, top_cases, llm, auto_config_override; при параметре детализации — массив trade_effects по сделкам. Дни, стратегия и флаг LLM из формы выше подставляются в ссылки и в кнопки скачивания.

Открыть в браузере (можно «Сохранить как…»):

  • Полный отчёт — как после «Запустить анализ»
  • То же + массив trade_effects по каждой сделке

Путь для curl / копирования:

curl -sS "https://ВАШ_ХОСТ/api/analyzer?days=7&strategy=GAME_5M&use_llm=0" -o report.json

Сводка периода (summary)

Сумма недоборов — sum_missed_upside_pct; сумма избежимых просадок по окну — sum_avoidable_loss_pct. Пояснения — в справке сверху.

Нажмите «Запустить анализ».

CatBoost на входе (бэктест) catboost_entry_backtest

Для GAME_5M и ALL (только сделки с entry_strategy=GAME_5M): та же модель, что при живом входе, но признаки только из сохранённого context_json на BUY — проверка благоприятности входа vs факт realized_pct. Опционально: корреляция estimated_upside_pct_day на входе с результатом. Портфельная игра — блок пропускается (другой контекст).

Худшие исходы (top_cases.top_losses)

Сначала сделки с наименьшим realized_pct (до 8 строк на странице). При высоком win rate это часто самые маленькие плюсы; смотрите также missed_upside_pct и exit_signal.

—

Крупнейший недобор — все сделки (top_cases.top_missed_upside)

Сортировка по убыванию missed_upside_pct (до 8 строк): максимальный «недожатый» запас до high 5m-окна, без фильтра по прибыли/убытку.

—

Прибыльные с крупным недобором (top_cases.top_profitable_missed_upside)

Только realized_pct > 0; на странице — до 8 строк с missed_upside_pct ≥ 0.25. Полный список — в JSON с include_trade_details=1 → фильтр по trade_effects.
